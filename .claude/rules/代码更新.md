# 文件位置规范

对于非项目本身的代码文件如:测试项目架构的代码脚本, 需要明确区分放在不同的文件夹中, 以免混淆小木核心代码以及测试脚本

# 代码更新规范

修改或者新增代码和代码文件时, 需要同步更新注释以及 `.claude\introduction\数据流\代码数据流.md` 文件的数据流说明, 以保证代码和文档的一致性和可维护性
此文档需要记录:

- 项目文件夹/文件结构说明
- 代码文件的功能说明
- 代码文件中每个函数的功能说明
- 代码文件中每个函数的输入输出说明
- 代码文件中每个函数的调用关系说明
- 代码文件中每个函数的调用时机说明
- 代码文件中每个函数的调用者说明
- 代码文件中每个函数的被调用者说明
- 代码文件中每个函数的调用流程说明

> 说明文件需要详略得当, 只确保每个模块的说明能够 **交给单个团队成员也能够对接已有模块了解其数据流并实现后续功能的开发** 的程度即可

# 修改代码同步更新修复记录

对于修复或更新功能需要同步写入: `.claude\introduction\修改记录` 文件夹中, 添加或修改md说明文档, **修改的说明文档名称需要与修改的文件命名相同(无后缀)**
主要记录:

- 修改的文件名和路径
- 修改前存在的问题(修改代码)
- 添加前未完成的功能(新增代码)
- 修改后的预期功能或修复后的预测结果

对于新增的修复记录文档,需要在claude.md的`# 代码修复记录`中添加`@`出对应文件如: `@introduction\修改记录\文件名.md` 以便于查看和追踪

# 架构实现流程规范

本项目不应该一次性完成所有代码实现, 而是应该确保每次代码都形成一个可验证的完整多模块的功能联动的实现后, 再进行后续项目可进行的功能的实现和开发, 分步骤迭代
