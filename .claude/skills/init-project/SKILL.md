当项目为空文档说明的项目（目标边界、项目目标、STEP.md 为模板空内容）时，先判断文档状态，再委托 `update-docs` agent 读取项目文档和项目代码，分析并更新项目文档。更新范围取决于项目是否有代码：

- **项目完全空（无代码，环境说明/数据流/STEP 等均为空模板，只有用户管理的项目文档）**：只更新目标边界、项目目标、STEP.md 三个文件。
- **项目已有代码**：除上述三个文件外，还需更新环境说明文档（本地开发环境.md、常用命令.md）和数据流文档（核心数据流.md）。

**代码是否存在的判断由 `update-docs` agent 负责，你不自行判断。**

若既没有项目文档也没有代码，则由 `update-docs` agent 报告缺乏项目背景信息。

## 你的职责

- 只为判断项目是否处于空项目状态读取三个目标文件（三个目标文件均为模板空内容）。
- 允许只为判断项目文档是否存在，检查 `.claude_introduction/项目文档/` 下是否有 `.md` 文档；不要为了补充 prompt 主动阅读、搜索或分析项目文档内容。
- 不主动阅读、搜索或分析项目代码；代码阅读、代码搜索、项目文档阅读、项目背景提炼和变更细节确认都交给 `update-docs` agent。
- 基于已有上下文与允许读取的文档状态，委托 `update-docs` agent 更新文档；更新范围由 `update-docs` agent 根据项目是否有代码动态决定（见上方规则）。
- 不直接手写目标文件内容；由 `update-docs` agent 按既定格式写入。
- 完成后不要求自动创建 git commit（初始化阶段由用户决定是否提交）。

## 触发条件

以下三个文件**全部**为模板空内容时触发：

- `.claude_introduction/项目实现目标/目标边界.md`
- `.claude_introduction/项目实现目标/项目目标.md`
- `.claude_introduction/TODO/STEP.md`

判断标准：文件内容仅包含标题结构，无实质性项目事实（如 `待补充`、`[任务标题]`、空白字段）。

## 三个目标文件的功能定位

分析项目文档时，按以下分工将提取的信息映射到对应文件：

### 目标边界.md — 阶段边界与验收约束

定义当前阶段的**范围和硬约束**，回答"这一阶段要做到什么、不做什么、做到什么程度算合格"。

| 字段 | 含义 | 信息来源 |
|------|------|----------|
| 总体目标 | 项目最终要达成的完整目标（一句话） | 项目文档中的项目介绍/背景 |
| 当前阶段目标 | 本阶段具体要完成的子目标 + 验收方式 | 项目文档中的阶段划分/里程碑 |
| 成功标准 | 可验证的完成标准 | 项目文档中的验收条件/交付物要求 |
| 非目标 | 明确不属于当前阶段的内容 | 项目文档中的范围排除/后续阶段内容 |
| 质量底线 | 测试/安全/性能/文档的硬性要求 | 项目文档中的质量标准/合规要求 |
| 推进节奏 | 阶段推进规则（如"每阶段需可验证交付"） | 项目文档中的开发流程/迭代规则 |

### 项目目标.md — 功能范围与实现追踪

定义**要做什么功能**以及实现过程中的**发现和调整**，随开发推进持续更新。

| 字段 | 含义 | 信息来源 |
|------|------|----------|
| 总体流程目标 | 完整流程大方向 | 项目文档中的整体愿景/最终形态 |
| 核心功能目标 | 需要实现的功能 + 验收方式（checklist） | 项目文档中的功能列表/需求描述 |
| 阶段功能范围 | 按阶段/模块拆分，含目标、状态、依赖 | 项目文档中的分期规划/模块划分 |
| 实现发现 | 开发过程中发现的目标/边界变化（初始写"暂无"） | 初始为空，由 update-docs 后续维护 |
| 后续目标调整 | 待确认或后续推进的目标（初始写"暂无"） | 初始为空，由 update-docs 后续维护 |

### STEP.md — 长期方向与阶段步骤

定义**大方向路线图**，不随每次提交更新，只在方向性调整时修改。

| 字段 | 含义 | 信息来源 |
|------|------|----------|
| 长期方向 | S-xxx 编号的大方向步骤，含目标、完成标准、影响范围 | 项目文档中的路线图/长期规划 |

注意：STEP.md 只记录长期大步骤，细粒度任务在 TODO.md 中维护。

## 各文件格式模板

具体格式见 `references/file-formats.md`；写入时由 `update-docs` agent 按其既定格式模板执行，不自由发挥。

## 执行流程

### 1. 确认空项目状态

先读取三个目标文件，确认均为空模板。如果任一文件已有实质内容，停止初始化流程并报告：
- 哪些文件已有内容
- 建议使用 `update-docs` skill 做增量更新

### 2. 确认项目背景来源存在

检查 `.claude_introduction/项目文档/` 下是否存在 `.md` 文件。此步骤只判断文档是否存在，不读取和分析文档内容。

如果目录为空或无 `.md` 文件，仍可继续委托 `update-docs` agent 检查项目代码是否足以初始化；如果既没有项目文档也没有可用代码信息，由 `update-docs` agent 报告缺少项目背景信息。

### 3. 委托 update-docs agent 阅读并分析

不要在当前 skill 中主动阅读、搜索或分析项目代码，也不要为了补充 prompt 主动读取项目文档内容。将以下工作交给 `update-docs` agent：

- 读取 `.claude_introduction/项目文档/` 下所有 `.md` 文件（若存在）。
- 阅读和搜索项目代码（若需要）。
- 提取项目背景、目标、范围、阶段、约束、验收标准等信息。
- 判断信息是否足以初始化三个目标文件。

### 4. 委托 update-docs agent 更新

调用 `update-docs` agent，在 prompt 中明确：

- **只更新以下三个文件，不修改其他文档**：
  - `.claude_introduction/项目实现目标/目标边界.md`
  - `.claude_introduction/项目实现目标/项目目标.md`
  - `.claude_introduction/TODO/STEP.md`
- 说明当前已确认三个目标文件均为空模板。
- 说明当前是否存在 `.claude_introduction/项目文档/` 下的 `.md` 文件（只提供存在性判断，不提供自行提炼内容）。
- 要求 agent 自行读取必要项目文档与项目代码，完成初始化分析。
- 明确要求**不创建 git commit**（初始化阶段由用户自行决定提交时机）。
- 要求 agent 按既定格式模板填写，不自由发挥格式。

推荐 prompt 模板：

```md
## 任务：初始化项目核心规划文件

当前 init-project skill 已确认：
- `.claude_introduction/项目实现目标/目标边界.md` 是空模板
- `.claude_introduction/项目实现目标/项目目标.md` 是空模板
- `.claude_introduction/TODO/STEP.md` 是空模板
- `.claude_introduction/项目文档/` 下是否存在 `.md` 文件：[是/否]

请你自行读取必要的项目文档和项目代码，分析当前项目背景、目标、范围、阶段、约束与验收标准，并基于分析结果更新三个目标文件。

### 只更新以下文件
- `.claude_introduction/项目实现目标/目标边界.md`
- `.claude_introduction/项目实现目标/项目目标.md`
- `.claude_introduction/TODO/STEP.md`

### 不修改的文件
- `.claude_introduction/项目文档/` 下所有文件（只读）
- `.claude_introduction/环境说明/` 下所有文件
- `.claude_introduction/数据流/` 下所有文件
- `.claude_introduction/TODO/TODO.md`
- 其他一切文件

### 分析要求
- 优先读取 `.claude_introduction/项目文档/` 下已有项目文档（若存在）
- 必要时阅读和搜索项目代码补充事实
- 如果项目文档和代码都不足以支撑初始化，请停止并报告缺少哪些背景信息
- 不要臆造项目事实；无法确认的内容写为待确认或暂无

### 要求
- 严格按照 update-docs agent 既定的文件格式模板填写
- 不创建 git commit
- 返回修改了哪些文件、每类修改的核心内容摘要
```

### 5. 结果确认

`update-docs` agent 返回后，确认：
- 三个文件是否均已更新
- 格式是否符合模板要求
- 内容是否与项目文档一致
- 其他文件是否未被修改

## 与 update-docs 的区别

| 场景 | 使用 |
|------|------|
| 项目初始化（三个核心文件为空） | `init-project` |
| 增量更新（开发过程中文档变化） | `update-docs` |

`init-project` 是一次性初始化工具；项目进入正常开发迭代后，文档更新统一使用 `update-docs`。
