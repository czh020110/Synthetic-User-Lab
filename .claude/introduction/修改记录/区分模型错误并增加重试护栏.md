# 区分模型错误并增加重试护栏

## Commit

区分模型错误并增加重试护栏

## 变更目标

为 run graph 中的决策/验证 agent 增加模型 API 错误与结构化格式错误的重试护栏，避免首次调用失败就把测试任务直接判定为系统异常；同时把模型错误和系统异常在状态接口与最终报告中显式区分，保留原始错误信息，便于定位到底是测试链路问题还是模型调用问题。

## 修改前

决策与验证节点直接调用 agent，模型 API 报错或结构化回复格式错误时会立刻中断 run。失败记录只保留一类通用 `error_message`，状态接口和最终报告无法明确区分模型调用错误与系统异常，格式错误也没有通过补充提示词继续纠正的机会。

## 修改后

决策与验证 agent 现在共用一套重试逻辑：模型 API 错误最多重试 5 次，结构化格式错误最多追加格式修正提示词重试 3 次。重试仍失败时，run graph 会把错误统一标记为 `model_error`，并将原始 API 报错或格式校验报错继续写入 `RunRecord`、状态接口和最终报告；报告文案也会区分模型调用错误与系统异常中断。

## 修改文件

- `backend/graph/run_graph.py`
  - 修改原因：需要在决策/验证节点之间复用模型调用重试护栏，并统一抛出可区分的模型错误。
  - 修改内容：新增模型调用重试 helper、`ModelInvocationError`、格式重试消息组装逻辑，并让 `decide_next_action()`、`validate_current_progress()` 走同一套 API 错误与格式错误重试流程；`run_workflow()` 的异常兜底改为区分 `model_error` 与 `system_error`。
  - 修改影响：run graph 不会在首次模型调用失败时直接中断，失败收尾链路可向状态接口和报告传递更准确的错误语义。

- `backend/prompt/graph.py`
  - 修改原因：格式错误重试提示词需要放在独立 prompt 模块中，不能内嵌到 graph 实现里。
  - 修改内容：新增 `format_retry_input`，约束模型在格式修正重试时只按指定 response format 与 JSON Schema 返回结构化结果。
  - 修改影响：格式修正文案与 graph 实现解耦，后续调整提示词时不必改运行节点逻辑。

- `backend/schemas/run_schemas.py`
  - 修改原因：运行状态与最终报告需要显式暴露错误类型和原始错误信息。
  - 修改内容：新增 `RunErrorType`，并在 `RunRecord`、`RunStatusResponse`、`RunReport` 中补充 `error_type` / `error_message` 字段。
  - 修改影响：调用方可以直接区分模型错误与系统异常，避免把所有失败都归入同一类任务错误。

- `backend/stores/in_memory_run_store.py`
  - 修改原因：内存存储需要持久化新的错误类型字段。
  - 修改内容：`fail_run()` 支持写入 `error_type`，状态查询返回 `error_type`，成功收尾时会清理旧错误信息。
  - 修改影响：run 状态轮询与异常兜底报告可以读取一致的错误类型数据。

- `backend/analysis/report_builder.py`
  - 修改原因：最终报告需要带出错误类型，并在关键发现中区分模型调用错误和系统异常。
  - 修改内容：`build_run_report()` 输出 `error_type` / `error_message`；关键发现文案改为按错误类型生成“模型调用错误”或“系统异常中断”描述，并把错误类型传给建议生成上下文。
  - 修改影响：失败报告更贴近真实根因，便于后续按模型问题或系统问题分别处理。

- `tests/test_validator.py`
  - 修改原因：需要为新增的重试与错误分类逻辑提供回归保护。
  - 修改内容：补充 agent API 错误重试、格式错误重试、格式错误超限以及 `model_error` 状态/报告兜底测试。
  - 修改影响：后续调整 run graph 时，若破坏模型错误收尾或格式重试边界会被测试及时发现。

- `.claude/introduction/数据流/核心数据流.md`
