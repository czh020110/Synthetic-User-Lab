# 调整任务完成判定以收敛报告边界

## Commit

调整任务完成判定以收敛报告边界

## 变更目标

让当前 run 的成功判定更明确地围绕 persona 驱动的任务执行结果展开，同时把最终报告的建议边界收紧到本次 run 的真实过程、摩擦信号和 persona 行为特征，避免把项目路线图式建议写进运行报告。

## 修改前

任务成功判定仍偏向于依赖页面中的固定成功文案，报告边界也没有被明确写入阶段目标与数据流说明，容易让后续实现继续把项目规划、技术债或长期路线图混入 run 级报告建议。

## 修改后

当前 validator 与 graph prompt 已按“任务描述 + success_criteria + 真实页面状态 + 执行结果”的组合来判断完成态，并允许 validate agent 直接给出 succeeded 结论；目标边界补充了报告底线和 persona 底线，明确最终报告的建议必须基于本次 run 的真实执行过程动态生成，核心数据流也同步说明了 validate 与 finalize_report 的真实职责。

## 修改文件

- `backend/analysis/validator.py`
  - 修改原因：需要让成功判定与当前任务目标保持一致，减少对固定成功文案的依赖。
  - 修改内容：调整 validate 结果的归一化逻辑，允许 agent 明确给出的 succeeded 结论直接进入成功态，并保持失败、卡住、偏航等规则护栏分支的统一入口。
  - 修改影响：run 级验证结果更容易直接结束成功路径，后续报告生成会基于实际完成态收尾。

- `backend/graph/demo_run_graph.py`
  - 修改原因：demo 任务需要更贴近“验证当前实现是否可用”的语义，而不是只依赖某个固定成功文案。
  - 修改内容：更新 demo task 的名称、描述与 success_criteria，使任务成功标准更偏向页面明确完成态而不是单一文本命中。
  - 修改影响：demo 场景的任务定义更适合当前验证闭环。

- `backend/graph/graph_prompt.py`
  - 修改原因：决策与验证提示需要跟随新的成功判定边界。
  - 修改内容：补充 validate 提示中对 task.description、success_criteria、最新页面状态与动作结果的综合判断说明，并要求成功时直接输出 succeeded。
  - 修改影响：agent 侧的成功判断更贴近当前任务语义。

- `tests/test_validator.py`
  - 修改原因：需要回归验证新的成功判定与 graph 节点行为。
  - 修改内容：调整 success 场景、agent success 场景与 graph 层断言，确保 validator 与 demo graph 按新边界工作。
  - 修改影响：为成功判定变化提供最小自动化保障。

- `.claude/introduction/项目实现目标/目标边界.md`
  - 修改原因：需要把最终运行报告的边界原则写入阶段约束。
  - 修改内容：新增报告底线与 persona 底线，明确报告建议必须基于本次 run 的真实过程和 persona 行为特征，不写项目路线图式建议。
  - 修改影响：后续实现和验收时可直接按该边界约束 report 输出。

- `.claude/introduction/数据流/核心数据流.md`
  - 修改原因：validate 与 finalize_report 的真实职责发生了收敛，需要同步到核心数据流。
  - 修改内容：更新 validate_current_progress、finalize_report、RunReport 输出用途和错误分支描述，补充 report_builder 相关文件引用。
  - 修改影响：接手者可按当前真实链路理解成功判定与最终报告的职责边界。

- `.claude/introduction/修改记录/调整任务完成判定以收敛报告边界.md`
  - 修改原因：需要按本次 commit 维度沉淀修改记录。
  - 修改内容：记录本次任务完成判定与报告边界同步的原因、范围和影响。
