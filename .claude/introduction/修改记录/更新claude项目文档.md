# 更新claude项目文档

## Commit

更新claude项目文档

## 变更目标

把项目文档体系拆分为更稳定的 introduction 结构，并明确文档查询、更新和 TODO 维护边界。

## 修改前

项目文档、规则、技能说明和进度文档的边界不够清晰，后续维护容易混杂长期背景、当前阶段目标和一次提交粒度任务。

## 修改后

形成以 `introduction/` 为核心的项目文档结构，补齐目标边界、项目目标、本地环境、核心数据流、STEP/TODO 等固定入口，并同步 query-project 与 update-docs 的使用说明。

## 修改文件

- `.claude/CLAUDE.md`
  - 修改原因：需要为项目文档提供统一入口。
  - 修改内容：整理项目文档、环境、TODO、数据流与技能文档的引用。
  - 修改影响：后续查询与维护路径更明确。
- `introduction/项目文档/*`
  - 修改原因：需要把长期项目设计拆分为分块文档。
  - 修改内容：按定位边界、模块接口、RAG、架构、LangGraph、MVP 路线等主题拆分文档。
  - 修改影响：项目长期背景可按主题检索与维护。
- `introduction/项目实现目标/目标边界.md`、`introduction/项目实现目标/项目目标.md`
  - 修改原因：需要独立维护阶段目标与整体目标。
  - 修改内容：补齐总体目标、当前阶段目标、成功标准、非目标、质量底线与阶段范围。
  - 修改影响：文档边界与验收口径更清楚。
- `introduction/环境说明/本地开发环境.md`、`introduction/环境说明/常用命令.md`
  - 修改原因：需要集中记录环境与命令事实。
  - 修改内容：沉淀本地环境要求、推荐端口和常用启动命令。
  - 修改影响：后续开发与验证入口更统一。
- `introduction/TODO/STEP.md`、`introduction/TODO/TODO.md`
  - 修改原因：需要区分长期路线与下一次提交任务。
  - 修改内容：引入 STEP 与 TODO 的双层维护方式。
  - 修改影响：计划拆分粒度更合理。
- `.claude/skills/query-project/*`、`.claude/skills/update-docs/*`
  - 修改原因：需要让项目文档检索与更新规则可执行。
  - 修改内容：补充技能说明、脚本与引用文档。
  - 修改影响：后续通过技能检索和维护文档的路径更稳定。
