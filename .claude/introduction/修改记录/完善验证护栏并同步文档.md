# 完善验证护栏并同步文档

## Commit

完善验证护栏并同步文档

## 变更目标

把 demo run 的验证节点从单一 agent 判定升级为“agent 结果 + 规则护栏”双层收敛，并同步更新项目目标、数据流与进度文档，让停止条件、恢复候选和最终报告文案保持一致。

## 修改前

验证节点主要依赖 agent 输出，停止条件、卡住、偏航和恢复候选边界没有统一的规则护栏收敛；最终报告摘要也没有直接使用最后一步的进展总结，项目文档中的阶段状态和数据流说明仍停留在旧表述。

## 修改后

验证节点先接收 `validate_agent` 输出，再由规则护栏统一补强成功、失败、卡住、偏航和恢复候选判断；最终报告摘要直接使用最后一步的 `progress_summary`。相关项目目标、目标边界、核心数据流以及 TODO / DONE 状态已同步更新。

## 修改文件

- `backend/analysis/validator.py`
  - 修改原因：把最小规则验证器升级为验证护栏，补齐重复等待、重复动作、卡住、偏航和恢复候选的统一判定入口。
  - 修改内容：增加基于历史步骤和当前动作的统计逻辑，支持对 agent 验证结果做二次收敛，并在阈值触发时返回硬停止结果。
  - 修改影响：`ValidationResult` 的 `progress_summary`、`friction_signals` 和 `should_stop` 现在同时受 agent 与规则层影响。

- `backend/graph/demo_run_graph.py`
  - 修改原因：让图层验证节点真正使用规则护栏，而不是只信任 agent 结构化输出。
  - 修改内容：在 `validate_current_progress()` 中先调用 `validate_agent`，再调用规则验证函数，最终只按合并后的 `should_stop` 进行路由。
  - 修改影响：`route_after_log` 的结束分支现在由护栏统一控制。

- `backend/analysis/report_builder.py`
  - 修改原因：让报告更贴近最后一步真实判定结果。
  - 修改内容：把报告摘要和关键发现中的最后一步结论改为直接使用 `progress_summary`。
  - 修改影响：失败报告和成功报告都更容易回放当前一步的判定原因。

- `tests/test_validator.py`
  - 修改原因：补齐验证护栏和图层接入后的回归覆盖。
  - 修改内容：新增重复等待、重复动作、卡住、偏航、恢复候选、agent 结果被规则覆盖、`route_after_log` 分支等测试。
  - 修改影响：可直接验证规则护栏不会被 agent 单点覆盖。

- `.claude/introduction/TODO/TODO.md`
  - 修改原因：轮转下一次提交的细粒度任务。
  - 修改内容：把本次完成的验证护栏工作从 TODO 迁出，并写入下一次提交可完成的新任务。
  - 修改影响：下一轮开发目标切换为最小恢复分支。

- `.claude/introduction/TODO/DONE.md`

  - 修改影响：进度文档与实际完成情况保持一致。

- `.claude/introduction/项目实现目标/项目目标.md`

  - 修改内容：把停止条件与任务验证增强标记为完成，并更新实现发现与后续目标。
  - 修改影响：项目目标从“验证与停止条件收敛中”切换为“准备推进失败恢复分支”。

- `.claude/introduction/项目实现目标/目标边界.md`
