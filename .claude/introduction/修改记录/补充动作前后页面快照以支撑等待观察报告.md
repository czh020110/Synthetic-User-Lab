# 补充动作前后页面快照以支撑等待观察报告

## Commit

补充动作前后页面快照以支撑等待观察报告

## 变更目标

让等待观察与普通动作的步骤日志不再只依赖单一页面快照或 `ExecutionResult` 附带字段，而是统一保留动作前页面、动作后页面和等待观察逐轮截图，支撑 `/steps` 输出与最终报告提供更完整的页面证据链。

## 修改前

步骤日志只保留单一 `observed_page_state`，动作执行器同时承担截图和动作后 URL 回传职责。等待观察虽然已有 `wait_observation_*` 结果，但逐轮 trace 没有截图路径，报告输入也只拆散序列化页面 URL、标题、摘要和报错，缺少动作前后页面对照。

## 修改后

通用 run graph 在动作执行前保存 `step-before` 截图，在非 wait 动作后通过 `observe_after_action` 记录 `step-after` 页面快照，在 wait 观察期间为每轮 trace 保存截图并在结束时记录最终动作后页面。`ObservedPageState` 统一承载截图路径，`StepLog` 显式区分动作前后页面状态，报告序列化改为输出 `before_page_state` 与 `after_page_state` 完整对象。demo fixture 的成功切换时长也延长到 30000ms，以稳定覆盖多轮等待观察回归。

## 修改文件

- `backend/graph/run_graph.py`
  - 修改原因：需要把截图与页面快照职责从执行结果中拆出，统一到 graph 的动作前、动作后和等待观察节点。
  - 修改内容：新增 `observe_after_action()`；`execute_current_action()` 在执行前保存 `step_before_page_state`；`wait_after_action()` 增加逐轮截图回调和最终动作后截图；`log_current_step()` 写入 `post_action_page_state`。
  - 修改影响：步骤日志改为同时保留动作前后页面证据，等待观察 trace 能关联到具体截图。
- `backend/graph/wait_observer.py`
  - 修改原因：需要让等待观察结果能携带逐轮截图与更真实的耗时。
  - 修改内容：`WaitObservationTrace` 新增 `screenshot_path`；`observe_until_ready()` 支持 `capture_trace_screenshot_fn`，并把每轮模型处理耗时计入等待预算。
  - 修改影响：等待观察轨迹可在步骤日志和报告中直接展示每轮截图证据。
- `backend/execution/playwright_adapter.py`
  - 修改原因：执行器不再适合作为页面证据聚合点。

- `backend/graph/run_state.py`
