# 统一等待观察链路以支撑模型化动态等待

## Commit

统一等待观察链路以支撑模型化动态等待

## 变更目标

把普通 `wait` 动作和等待观察节点收敛到同一条 wait 链路，让 wait model 决定每轮等待时长、等待类型和终止决策，并把等待轨迹、累计时长与异常超时结果稳定写入步骤日志和最终报告。

## 修改前

普通 `wait` 动作仍由执行器直接完成固定等待，等待观察节点只在 `recovery_candidate` 场景下单独触发；等待结果、累计时长、终止决策与 trace 没有统一沉淀到 run state、step log 和 report 中，普通等待与异常等待也没有明确的预算区分。

## 修改后

`wait_after_action` 成为统一入口：普通 `wait` 动作和验证后的等待观察都进入同一条 wait node，由快速模型返回 `normal_waiting` / `abnormal_stuck` / `ready_for_next_action` / `task_completed` 与 `next_wait_ms`。相关等待轨迹、累计等待时长、超时预算、terminal decision 与报告摘要都能稳定透出。

## 修改文件

- `backend/graph/run_graph.py`
  - 修改原因：需要把普通 `wait` 和等待观察统一到同一条受控等待链路。
  - 修改内容：调整图路由，让 `wait` 动作进入 `wait_after_action`；等待结束后再统一进入验证收尾；异常兜底补充浏览器关闭。
  - 修改影响：run graph 不再把 `wait` 当成普通执行器动作，而是由 wait node 统一控制等待节奏与结果。

- `backend/graph/wait_observer.py`
  - 修改原因：等待观察需要根据模型输出决定等待类型、时长和终止状态。
  - 修改内容：扩展等待决策与轨迹结构，区分正常等待、异常卡住、可继续操作和任务完成。
  - 修改影响：等待不再是固定 timeout，而是可被模型逐轮驱动的动态过程。

- `backend/execution/playwright_adapter.py`
