# 接入等待观察分支以支撑恢复路径

## Commit

接入等待观察分支以支撑恢复路径

## 变更目标

在现有验证护栏之上补一个最小等待观察分支，让 `recovery_candidate` 不再只是一个静态摩擦信号，而是可以先进入等待观察循环，判断页面是否已经进入可继续状态；同时把等待观察结果、快速模型配置和报告透出链路一起同步到文档与阶段计划中。

## 修改前

run graph 已经能够识别卡住、偏航和恢复候选，但 `recovery_candidate` 只会停留在验证结果里，没有独立的等待观察节点承接；步骤日志、最终报告和环境说明也没有明确描述等待观察结果、快速模型配置和相关数据流。

## 修改后

run graph 在验证后新增 `wait_after_action` 分支，并通过 `backend/graph/wait_observer.py` 里的等待观察循环，使用快速模型判断页面是继续等待、可以继续操作还是已经完成。等待结果会写入 `StepLog` 与最终 `RunReport`，`project goals`、`goal boundary`、`environment docs` 和 `core data flow` 也同步反映了这条新链路。

## 修改文件

- `backend/graph/run_graph.py`
  - 修改原因：需要在验证护栏命中 `recovery_candidate` 后接入等待观察节点，并把等待结果写回状态与步骤日志。
  - 修改内容：新增 `observe_after_action`、`wait_after_action`、`route_after_validate` 与等待超时验证分支；`validate_current_progress()` 支持复用/覆盖页面状态并在条件命中时进入等待观察；`log_current_step()` 记录等待结果字段。
  - 修改影响：run graph 不再只走 observe → decide → execute → validate 的单一路径，卡住候选可以先进入等待观察再决定是否继续。

- `backend/graph/wait_observer.py`
  - 修改原因：等待观察需要独立的循环、状态结果和轨迹记录结构。
  - 修改内容：新增 `WaitObservationDecision`、`WaitObservationOptions`、`WaitObservationTrace`、`WaitObservationResult` 与 `observe_until_ready()`。
  - 修改影响：等待观察逻辑从 run graph 中拆出，便于后续复用和测试。

- `backend/prompt/graph.py`
  - 修改原因：等待观察节点需要独立提示词，不能继续复用决策/验证提示词。
  - 修改内容：新增 `wait_observe` 与 `wait_observe_input`，约束模型只输出等待观察结论。
  - 修改影响：等待观察与动作决策、步骤验证的提示词职责分离。

- `backend/schemas/run_schemas.py`
  - 修改原因：步骤日志与报告需要保存等待观察结果。
  - 修改内容：为 `StepLog` 增加 `wait_observation_status`、`wait_observation_reason`、`wait_observation_observations`、`wait_observation_traces`。
  - 修改影响：等待观察轨迹可被报告和接口直接消费。

- `backend/graph/run_state.py`
  - 修改原因：图状态需要承载等待观察中间结果与 wait agent。
  - 修改内容：补充 `wait_observation_*` 字段与 `wait_agent`。
  - 修改影响：等待观察节点可以在 graph state 内保持上下文。

- `backend/execution/observer.py`
  - 修改原因：等待观察更依赖真实可见表单状态，需要过滤不可见字段。
  - 修改内容：表单字段收集增加可见性过滤。
  - 修改影响：页面观察结果更贴近真实可交互状态。

- `backend/execution/playwright_adapter.py`
  - 修改原因：受控动作执行需要更严格的参数校验。
  - 修改内容：`navigate` / `click` / `fill` 强制检查 `target`，`wait` 支持数值型等待时长。
  - 修改影响：降低等待观察之后重新执行动作时的异常输入风险。

- `backend/graph/demo_run_graph.py`
  - 修改原因：demo 场景需要把 wait agent 注入通用图执行链。
  - 修改内容：`load_demo_context()` 现在返回 `wait_agent`，并继续沿用 demo 默认 persona/task 装配逻辑。
  - 修改影响：demo run 可以直接使用等待观察链路。

- `backend/analysis/validator.py`
  - 修改原因：验证护栏需要识别停滞页面并给出 `recovery_candidate`，以便触发等待观察。
  - 修改内容：在重复等待、重复动作、卡住和偏航判断的基础上，继续输出可进入等待观察的候选信号。
  - 修改影响：等待观察分支有了明确的规则入口。

- `backend/analysis/report_builder.py`
  - 修改原因：最终报告需要解释等待观察过程和异常结束原因。
  - 修改内容：报告关键发现和建议会读取 `wait_observation_*` 字段，并区分模型错误、等待超时与系统异常。
  - 修改影响：报告能说明 run 在等待阶段发生了什么，而不是只保留最后一步结论。

- `backend/core/config.py`
  - 修改原因：等待观察节点需要单独的快速模型配置。
  - 修改内容：增加 `fast_model_name` 读取逻辑，并为 OpenAI / DashScope 提供快速模型环境变量。
  - 修改影响：等待观察节点可以和主决策/验证模型分离。

- `backend/ai_api/provider_router.py`

  - 修改影响：`get_settings()` 可以同时读取主模型与等待观察模型配置。
