# 系统架构、技术栈与存储职责

本文用于理解整体系统分层、推荐技术栈、PostgreSQL 与 Redis 的职责划分，以及建议目录结构。

## 如何使用本文

- 当任务只涉及本文主题时，优先只加载本文作为上下文。
- 本文保留原始设计文档中的关键实现要求，并补充少量阅读说明。
- 本次拆分只改变文档组织方式，不改变原有系统架构结论。

## 7\. 系统架构设计

## 7.1 总体架构

```markdown
Frontend Dashboard
↓
FastAPI
↓
Task Orchestrator
↓
LangGraph Workflow
↙ ↓ ↘
RAG Playwright Validator
Layer Executor / Analyzer
↘ ↓ ↙
PostgreSQL / Object Storage / Redis
```

---

## 7.2 分层说明

## 7.2.1 前端层

职责：

- 创建测试任务
- 配置 persona
- 查看实时执行过程
- 回放步骤
- 查看截图
- 查看最终报告
- 比较不同版本 run

建议技术：

- Next.js / React
- ECharts / D3
- React Flow（可选）

---

## 7.2.2 API 层

职责：

- 提交任务
- 查询任务状态
- 获取 run 结果
- 管理 persona / task / project
- 暴露报告接口

建议技术：

- FastAPI
- Pydantic

---

## 7.2.3 编排层

职责：

- 管理 LangGraph 工作流
- 控制节点流转
- 处理异常分支
- 管理执行中断与恢复

建议技术：

- LangGraph
- LangChain（仅作工具与 retriever 封装）

---

## 7.2.4 执行层

职责：

- 与浏览器交互
- 执行动作
- 收集页面状态
- 截图与日志记录

建议技术：

- Playwright

---

## 7.2.5 记忆与检索层

职责：

- 产品知识检索
- UI 经验检索
- 失败案例检索
- 规则检索

建议技术：

- pgvector / Qdrant / FAISS
- LangChain retriever 封装
- PostgreSQL 作为元数据存储

---

## 7.2.6 数据与状态层

职责：

- 持久化 run、step、report、persona、task
- 缓存高频运行状态
- 存储截图与原始日志

建议技术：

- PostgreSQL：正式业务数据
- Redis：运行时缓存 / 队列 / 实时事件
- S3 / MinIO / 本地文件系统：截图与附件

## 可观测性

- **LangSmith** （可选）

- **Structured Logging**

---

## 10.2 各技术选型理由

## FastAPI

适合作为服务层：

- 类型友好
- OpenAPI 自动文档
- 易与 Pydantic 结合
- 适合异步任务 API

## LangGraph

适合管理：

- 有状态任务执行
- 多节点循环
- 异常恢复分支
- 中断与恢复

## Playwright

适合第一阶段 Web 产品自动化：

- 稳定
- 生态成熟
- 支持截图与元素操作
- 支持自动等待和可重试

## PostgreSQL

负责正式持久化：

- 项目、任务、run、step、report
- 适合查询分析与后期统计

## Redis

不是 MVP 硬前提，但平台化后非常有价值：

- 异步任务队列
- 实时状态缓存
- Pub/Sub 事件流
- 浏览器 session 状态协调

## 11\. Redis 与 PostgreSQL 的职责划分

## PostgreSQL

适合存：

- Project
- Persona
- Task
- Run
- StepLog
- Report
- FrictionIssue
- 知识元数据

## Redis

适合存：

- 当前 run 实时状态
- 流式事件
- worker 队列
- 临时页面观察缓存
- 最近 N 步上下文
- 浏览器会话映射
- 锁与去重信息
