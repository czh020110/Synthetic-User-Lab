## Synthetic User Lab 项目设计说明文档

## 1\. 项目概述

### 1.1 项目名称

**Synthetic User Lab**  
中文可译为： **合成用户实验室**

### 1.2 项目定位

Synthetic User Lab 是一个面向 **Web 应用 / AI 应用 / SaaS 产品** 的 **多智能体任务型用户测试平台** 。  
它通过构造不同用户画像（新手、专家、焦躁型、误解型、对抗型等），让这些合成用户在受控环境中自动执行真实任务流程，并基于任务结果、交互路径、错误恢复行为和摩擦点，输出结构化的产品可用性分析报告。

### 1.3 核心价值

它不是普通的：

它的核心价值在于：

1. **模拟不同用户类型的使用行为**
2. **在真实产品界面中执行任务**
3. **识别流程摩擦、误导性文案、恢复困难、AI 产品误用风险**
4. **沉淀产品知识、UI 操作经验和历史失败记忆**
5. **将“可用性测试 + AI 产品行为评测 + 经验记忆”统一起来**

---

## 2\. 项目目标与边界

## 2.1 项目目标

本项目旨在解决以下问题：

- 新用户是否能顺利完成关键任务
- 不同类型用户在哪一步最容易迷失
- 某些按钮、文案、流程是否有歧义
- AI 产品是否会引导用户产生错误预期
- 遇到错误时，用户是否能够恢复
- 某个功能是否在多版本迭代中变得更难用

---

## 2.2 非目标

本项目 **不追求** 以下能力：

- 不追求替代真人可用性测试
- 不承诺对任意软件进行稳定自动操作
- 不以“像真人一样自由玩电脑”为目标
- 不以传统问答型 RAG 为主功能
- 不做全平台万能 UI 自动化工具

---

## 2.3 第一阶段边界

为了保证可落地性，第一阶段明确限定为：

### 支持范围

- **Web 应用**
- **AI 对话类产品**
- **表单 / 设置 / 上传 / 导出 / onboarding 流程**
- **常规多步任务**

### 暂不支持

- 桌面重图形软件
- 移动端复杂原生应用
- 高度依赖拖拽的复杂交互
- CAD / PS / 游戏类软件
- 高风险真实支付 / 删除 / 外部副作用操作

---

## 3\. 典型使用场景

## 3.1 SaaS 新手 onboarding 测试

任务示例：

- 注册账号
- 创建第一个项目
- 上传示例文件
- 完成第一次配置
- 获取结果输出

目标：

- 判断新手能否独立完成
- 识别引导链路中的卡点
- 发现术语理解门槛

---

## 3.2 AI 产品任务完成测试

任务示例：

- 上传 PDF 并获取摘要
- 创建知识库并提问
- 调用某个 AI 工具完成指定操作
- 使用 agent 自动生成报告

目标：

- 判断 AI 产品的功能链路是否清晰
- 检查 AI 输出与用户预期是否匹配
- 识别工具调用不透明、反馈不足等问题

---

## 3.3 回归测试与版本对比

任务示例：

- 对比 v1 与 v2 在同一组任务上的表现差异
- 比较不同文案、不同按钮位置、不同 onboarding 策略对完成率的影响

目标：

- 做产品迭代前后的可用性对比
- 验证改动是否真的降低摩擦

---

## 3.4 安全与误用测试

任务示例：

- 对抗型用户尝试误用 AI 功能
- 焦躁型用户重复点击、乱跳路径
- 含糊型用户输入不完整任务

目标：

- 识别错误恢复弱点
- 发现文案歧义
- 发现易误触路径与危险操作暴露

---

## 4\. 核心设计理念

## 4.1 任务型而非自由型

合成用户不是无限自由探索，而是在 **任务约束** 下行动。  
每个 run 都必须绑定：

- 明确任务目标
- 起始页面
- 成功标准
- 最大步数
- 容忍失败次数

这样系统才可评估、可复现、可比较。

---

## 4.2 实时事实优先

当前界面状态必须来自真实环境观察，而不是模型臆测。  
系统决策必须优先依据：

- 当前 URL
- DOM 摘要
- 页面可交互元素
- 截图
- 当前报错
- 最近执行历史

---

## 4.3 RAG 作为上下文供给系统

RAG 在这里不是做知识问答，而是提供：

- 产品知识
- UI 经验
- 历史失败恢复经验
- 评测规则
- Persona 行为先验

---

## 4.4 经验可积累

每次 run 都应反哺系统：

- 哪些路径高成功率
- 哪些页面常导致迷失
- 哪些恢复动作有效
- 哪些 persona 最容易失败

这样系统不是静态脚本，而是 **会成长的体验实验室** 。

---

## 4.5 受控执行而非自由命令

LLM 不直接输出任意可执行代码。  
系统只允许 LLM 在受控 action schema 中选择动作，例如：

- click
- fill
- press
- upload
- scroll
- wait
- navigate
- ask_for_help
- abandon

这样能显著降低不确定性和安全风险。

---

## 14\. 关键注意事项

## 14.1 不要把它做成普通自动化测试框架

Selenium / Playwright 本身已经能做自动化测试。  
你的差异化不在“能点按钮”，而在：

- persona 驱动
- RAG 记忆支持
- 摩擦分析
- 失败恢复推理
- 多轮任务评估

---

## 14.2 不要让 LLM 直接操控原始命令

必须通过结构化 action schema 中转，否则不可控。

---

## 14.3 不要让 RAG 盖过实时页面事实

页面当前状态必须优先于历史知识。

---

## 14.4 不要一开始就做全平台

先把 Web 做稳。  
Web 足以验证这个项目的核心价值。

---

## 14.5 注意报告可解释性

不能只输出“这个页面不好用”。  
必须说明：

- 哪一步出问题
- 为什么出问题
- 哪类用户更容易中招
- 证据是什么
- 优先修什么

---

## 16\. 项目亮点总结

## 与普通 RAG 的区别

不是知识问答，而是 **任务执行时的上下文与经验记忆系统** 。

## 与普通 UI 自动化的区别

不是规则脚本，而是 **persona 驱动的任务型智能测试** 。

## 与通用 browser agent 的区别

不是泛化“能不能浏览网页”，而是 **可评估、可复现、可分析的产品体验实验系统** 。

## 与传统可用性测试的区别

不是替代真人，而是 **低成本筛掉明显问题，聚焦真人测试资源** 。

## 20\. 最后结论

这个项目是 **成立的，而且很适合做成一个真正有辨识度的 agent 项目** 。  
它最关键的三点是：

1. **把范围限定为 Web / AI 产品**
2. **把 RAG 变成经验与上下文系统，而不是问答库**
3. **把“点击网页”升级成“任务完成 + 摩擦诊断 + 可解释报告”**

这样它就不是一个“能操作浏览器的 LLM demo”，而是一个真正有产品和研究价值的系统。

你下一步最该做的是： **先定 MVP，只做 1 个产品、3 个 persona、3 个任务模板，然后把整条链路跑通。**
