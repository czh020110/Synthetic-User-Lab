## 2. 最小 Run 闭环模块整体目标

当前代码实现的是一个 **最小可运行纵向切片**：

1. FastAPI 暴露 demo run 接口
2. 接口启动一次 run
3. LangGraph 编排页面观察、动作决策、动作执行、结果验证、步骤记录、报告生成
4. Playwright 操作内置 Demo 页面
5. 运行状态、步骤日志、最终报告保存在内存中
6. 调用方可以查询 run 状态、steps、report

当前闭环只覆盖固定 demo persona、固定 demo task、固定 demo 页面，不包含数据库、RAG、前端管理页面和复杂恢复分支。

## 3. 整体数据流说明

## 3.1 主调用链路

```text
HTTP POST /api/v1/runs/demo/start
→ backend.api.routes.demo_runs.start_demo_run()
→ 创建 RunRecord 占位记录
→ asyncio.create_task(run_demo_workflow(...))
→ backend.graph.demo_run_graph.run_demo_workflow()
→ LangGraph 执行各节点
   1. load_demo_context
   2. init_session
   3. observe_page
   4. decide_action
   5. execute_action
   6. validate_progress
   7. log_step
   8. route_after_log
   9. finalize_report
→ backend.stores.in_memory_run_store 保存状态 / steps / report
→ HTTP GET /api/v1/runs/demo/{run_id}
→ HTTP GET /api/v1/runs/demo/{run_id}/steps
→ HTTP GET /api/v1/runs/demo/{run_id}/report
```

## 3.2 页面侧数据流

```text
backend/fixtures/demo_site/index.html
→ 用户点击“开始体验”
→ 显示表单
→ 用户填写 user_name / user_email
→ 点击“提交体验表单”
→ 若为空则显示错误提示
→ 若完整则隐藏表单并显示“提交成功”
```

## 3.3 运行态数据流

```text
RunRequest
→ DemoRunState
→ StepLog[]
→ RunReport
```

## 4.3 `backend/api/routes/demo_runs.py`

### 功能

暴露最小 Run 闭环的 HTTP 查询面。

### 当前接口

#### `health_check()`
- 路径：`GET /runs/demo/health`
- 输入：无
- 输出：`ApiResponse`
- 调用时机：健康检查、连通性验证
- 调用者：测试或外部调用方
- 作用：确认 FastAPI 服务可用

#### `start_demo_run(request: Request, payload: RunRequest)`
- 路径：`POST /runs/demo/start`
- 输入：
  - `Request`：用于获取当前服务 base URL
  - `RunRequest`：run_name、expected_user_name、expected_email、headless、operator_note
- 输出：`ApiResponse`，其中 `data` 包含 `run_id` 和 `queued` 状态
- 调用时机：用户启动一次 demo run 时
- 调用者：HTTP 客户端
- 被调用者：
  - `get_settings()`
  - `run_store.create_run()`
  - `run_demo_workflow()`
- 调用流程：
  1. 生成 `run_id`
  2. 根据当前请求生成 `app_base_url`
  3. 创建占位 `RunRecord`
  4. 写入内存存储
  5. 通过 `asyncio.create_task(...)` 异步启动 LangGraph run
  6. 立即返回 queued

#### `get_demo_run_status(run_id: str)`
- 路径：`GET /runs/demo/{run_id}`
- 输入：`run_id`
- 输出：`ApiResponse[RunStatusResponse]`
- 调用者：轮询 run 状态的客户端
- 被调用者：`run_store.get_status()`

#### `get_demo_run_steps(run_id: str)`
- 路径：`GET /runs/demo/{run_id}/steps`
- 输入：`run_id`
- 输出：`ApiResponse[list[StepLog]]`
- 调用者：查看步骤日志的客户端
- 被调用者：`run_store.get_steps()`

#### `get_demo_run_report(run_id: str)`
- 路径：`GET /runs/demo/{run_id}/report`
- 输入：`run_id`
- 输出：`ApiResponse[RunReport]`
- 调用者：查看最终报告的客户端
- 被调用者：`run_store.get_report()`、`run_store.get_record()`
- 特殊行为：
  - 不存在 run 返回 404
  - run 存在但 report 未生成返回 409

## 4.7 `backend/graph/demo_run_graph.py`

### 功能

该文件是最小 Run 闭环的核心编排层，负责将 API 输入变为完整执行流程。

### 关键函数

#### `load_demo_context(state: DemoRunState)`
- 输入：初始 state
- 输出：`persona`、`task`、`record`、`step_logs`、`current_step_index`、`should_stop`
- 调用时机：图开始后第一个节点
- 调用者：LangGraph
- 被调用者：`get_settings()`、`run_store.create_run()`、`run_store.mark_running()`
- 作用：
  1. 构造固定 demo persona
  2. 构造固定 demo task
  3. 创建或补齐运行记录
  4. 将 run 标记为 running

#### `init_session(state: DemoRunState)`
- 输入：含 task、request 的 state
- 输出：`session`
- 调用时机：persona / task 加载后
- 调用者：LangGraph
- 被调用者：`create_browser_session()`
- 作用：
  1. 根据 request/headless 计算浏览器模式
  2. 启动浏览器
  3. 打开任务起始 URL

#### `observe_current_page(state: DemoRunState)`
- 输入：`session`
- 输出：`current_page_state`
- 调用时机：每轮动作决策前
- 调用者：LangGraph
- 被调用者：`observe_page()`

#### `decide_next_action(state: DemoRunState)`
- 输入：`current_page_state`、`request`
- 输出：`current_action`
- 调用时机：页面观察后
- 调用者：LangGraph
- 作用：根据页面元素和字段值选择下一步动作
- 当前决策规则：
  1. 若存在 `#start-demo`，点击开始体验
  2. 若 `#user-name` 为空，填写用户名
  3. 若 `#user-email` 为空，填写邮箱
  4. 若存在 `#submit-demo`，点击提交
  5. 否则执行 wait

#### `execute_current_action(state: DemoRunState)`
- 输入：`session`、`current_action`、`run_id`、`screenshot_dir`
- 输出：`current_execution_result`
- 调用时机：动作决策后
- 调用者：LangGraph
- 被调用者：`execute_action()`
- 作用：
  1. 计算当前步骤截图路径
  2. 执行动作
  3. 记录动作结果和截图

#### `validate_current_progress(state: DemoRunState)`
- 输入：`session`、`task`、`current_execution_result`、历史 `step_logs`
- 输出：`current_page_state`、`current_validation_result`、`should_stop`
- 调用时机：动作执行后
- 调用者：LangGraph
- 被调用者：`observe_page()`、`validate_progress()`
- 作用：重新观察页面并判断成功、失败或继续

#### `log_current_step(state: DemoRunState)`
- 输入：当前页面状态、动作、执行结果、验证结果
- 输出：更新后的 `step_logs`、`current_step_index`
- 调用时机：验证后
- 调用者：LangGraph
- 被调用者：`run_store.add_step()`
- 作用：把本步写入内存步骤日志

#### `route_after_log(state: DemoRunState)`
- 输入：`should_stop`
- 输出：`"finalize_report"` 或 `"observe_page"`
- 调用时机：每一步写日志后
- 调用者：LangGraph 条件路由
- 作用：控制循环或结束

#### `finalize_report(state: DemoRunState)`
- 输入：`record`、`step_logs`
- 输出：`report`
- 调用时机：停止条件满足后
- 调用者：LangGraph
- 被调用者：`build_run_report()`、`run_store.complete_run()`、`close_browser_session()`
- 作用：
  1. 生成最终报告
  2. 写回内存存储
  3. 关闭浏览器会话

#### `build_demo_run_graph()`
- 输入：无
- 输出：编译后的 LangGraph 应用
- 调用时机：每次执行 run 前
- 调用者：`run_demo_workflow()`
- 作用：注册节点与边

#### `run_demo_workflow(run_id, request, app_base_url, screenshot_dir)`
- 输入：run 基础参数
- 输出：图执行最终 state
- 调用时机：`start_demo_run()` 异步启动时
- 调用者：`backend.api.routes.demo_runs.start_demo_run()`
- 被调用者：`build_demo_run_graph()`、`graph.ainvoke(...)`、`run_store.fail_run()`
- 作用：作为整个最小 run 闭环的异步入口

### 当前图流程

```text
START
→ load_demo_context
→ init_session
→ observe_page
→ decide_action
→ execute_action
→ validate_progress
→ log_step
→ route_after_log
   ├─ observe_page（继续循环）
   └─ finalize_report
→ END
```

## 4.8 `backend/execution/observer.py`

### 功能

把真实浏览器页面转换为结构化观察结果。

### 关键函数

#### `observe_page(page)`
- 输入：Playwright `page`
- 输出：`ObservedPageState`
- 调用时机：
  - 每轮决策前
  - 每轮执行后再验证前
- 调用者：`observe_current_page()`、`validate_current_progress()`
- 被调用者：页面内 JavaScript `evaluate(...)`
- 输出字段来源：
  - `current_url`：当前页面 URL
  - `title`：页面标题
  - `visible_text_summary`：正文压缩文本
  - `clickable_elements`：按钮/链接/role=button 摘要
  - `form_fields`：input/textarea/select 摘要
  - `error_messages`：可见错误提示摘要

### 调用流程

```text
page.title()
→ page.evaluate(读取页面文本)
→ page.evaluate(读取可点击元素)
→ page.evaluate(读取表单字段)
→ page.evaluate(读取可见错误提示)
→ 组装为 ObservedPageState
```

## 4.10 `backend/analysis/validator.py`

### 功能

用最小规则判断当前步骤是否成功、失败或继续。

### 关键函数

#### `validate_progress(task, observed_page_state, execution_result, previous_steps, current_step_index)`
- 输入：
  - `DemoTask`
  - `ObservedPageState`
  - `ExecutionResult`
  - 历史 `StepLog`
  - 当前步数
- 输出：`ValidationResult`
- 调用时机：每轮动作执行完成后
- 调用者：`validate_current_progress()`
- 当前判定逻辑：
  1. 动作执行失败 → `failed`
  2. 页面出现 `success_text` → `succeeded`
  3. 页面存在错误提示 → `failed`
  4. 当前步数达到上限 → `failed`
  5. 连续两次 `wait` → 记录 `repeated_wait`
  6. 其他情况 → `running`

- 调用者：`finalize_report()`

## 4.13 `backend/fixtures/demo_site/index.html`

## 7. 当前模块的边界与后续扩展点

## 8. 当前接手开发时的最小建议

1. 不要先拆掉最小闭环，先在现有闭环上加功能
2. 新增节点时优先复用 `DemoRunState` 的状态传递方式
3. 新动作优先先补 `ActionInput`、`execute_action()`、validator 判定
4. 将数据库接入时，先替换 `InMemoryRunStore`，不要先改图主流程
5. 外部真实网站接入前，优先保留 `demo_site` 作为回归验证页面
