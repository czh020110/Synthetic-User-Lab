# 核心数据流

## Demo Run 为等待观察链路补充动作前后页面快照与逐轮截图证据

### 本次更新原因

- 关联 Commit：补充动作前后页面快照以支撑等待观察报告
- 关联修改记录：`introduction/修改记录/补充动作前后页面快照以支撑等待观察报告.md`
- 更新原因：本次提交没有改动 demo API 入口和 wait node 主体路由，而是把单步日志与报告证据链从“单一页面快照 + execution_result 附带字段”改为“动作前页面快照 + 动作后页面快照 + 等待观察逐轮截图”。同时 demo fixture 的成功切换时长从 10 秒拉长到 30 秒，用于稳定覆盖多轮等待观察，因此需要同步更新 run graph、wait observer、StepLog、报告序列化与验证方式说明。

### 链路目标

在保留 `/runs/demo/*` 现有 API 入口和统一 wait node 的前提下，让每个 step 都能同时保留动作前页面事实、动作后页面事实，以及等待观察期间每一轮模型判断对应的截图证据，供 `/steps` 输出与最终报告直接消费；避免继续把截图路径和动作后 URL 混杂在 `ExecutionResult` 中，保证页面证据统一收敛到 `ObservedPageState` 与 `StepLog`。

### 修改前数据流

`POST /api/v1/runs/demo/start` 进入通用 run graph 后，`observe_page` 记录的是单一 `current_page_state`。`execute_action` 负责直接截图，并把 `screenshot_path`、`current_url_after_action` 写进 `ExecutionResult`；非 wait 动作执行后没有独立的动作后页面快照结构。等待观察链路虽然已经能输出 `wait_observation_*` 明细，但逐轮 trace 没有截图路径，报告序列化也只展开页面 URL、标题、文本摘要和错误信息，缺少动作前后页面对照证据。

### 修改后数据流

`POST /api/v1/runs/demo/start` → `backend.api.routes.demo_runs.start_demo_run()` → `backend.graph.demo_run_graph.create_demo_placeholder_record()` → `asyncio.create_task(run_demo_workflow(...))` → `backend.graph.demo_run_graph.run_demo_workflow()` → `backend.graph.run_graph.run_workflow(..., load_demo_context)` → `load_demo_context` → `init_session` → `observe_page` → `decide_action` → `execute_action(先保存 step-before)` → `wait_after_action(逐轮截图)` / `observe_after_action(保存 step-after)` → `validate_progress` → `log_step(写入 before/after page state)` → `finalize_report(序列化 before/after page state 与 wait trace 截图)` → `backend.stores.in_memory_run_store`

### 输入

- `RunRequest`
  - 来源：用户 / API
  - 关键字段：`run_name`、`headless`
  - 约束：由 `POST /runs/demo/start` 接收；`headless` 为空时回退到系统默认浏览器配置
- `app_base_url`
  - 来源：HTTP 请求
  - 关键字段：当前服务基础地址
  - 约束：由路由层从 `request.base_url` 生成，用于拼接 demo 起始页 `start_url`
- `ObservedPageState`
  - 来源：页面观察 / 代码内状态传递
  - 关键字段：`current_url`、`title`、`visible_text_summary`、`error_messages`、`screenshot_path`
  - 约束：动作前、动作后和等待结束后的页面快照都复用同一结构；截图路径统一挂在 `screenshot_path`，不再放在 `ExecutionResult`
- `WaitObservationOptions`
  - 来源：代码默认配置
  - 关键字段：`normal_timeout_ms`、`abnormal_timeout_ms`、`min_wait_ms`、`max_wait_ms`
  - 约束：wait model 每轮给出的 `next_wait_ms` 会被预算与阈值裁剪；每轮 trace 可额外捕获一张截图路径

### 处理步骤

1. `backend/api/routes/demo_runs.py::start_demo_run`
   - 职责：保留 demo API 入口，创建占位记录并异步启动后台 run
   - 调用方：HTTP 客户端
   - 被调用方：`get_settings()`、`create_demo_placeholder_record()`、`run_store.create_run()`、`run_demo_workflow()`
   - 调用时机：收到 `POST /runs/demo/start` 时
   - 输入：`Request`、`RunRequest`
   - 输出：包含 `run_id` 与 `queued` 状态的 `ApiResponse`
   - 状态变化：在内存存储中创建 `RunRecord(status=queued)`
   - 异常分支：创建后续流程失败时由后台 run 负责落失败状态；接口自身仅处理同步阶段异常

2. `backend/graph/demo_run_graph.py::create_demo_placeholder_record`
   - 职责：把 demo 默认 persona/task 组装到占位运行记录
   - 调用方：`start_demo_run()`
   - 被调用方：`build_demo_persona()`、`build_demo_task()`
   - 调用时机：路由层创建占位记录时
   - 输入：`run_id`、`RunRequest`、`app_base_url`
   - 输出：`RunRecord`
   - 状态变化：无额外副作用，仅返回待写入存储的记录
   - 异常分支：demo 默认上下文缺失时抛出异常，接口启动失败

3. `backend/graph/demo_run_graph.py::load_demo_context`
   - 职责：在真正执行前加载 demo 场景的默认 `Persona`、`Task`、agent 与运行记录
   - 调用方：`backend.graph.run_graph.build_run_graph()` 生成的 LangGraph
   - 被调用方：`build_demo_persona()`、`build_demo_task()`、`create_run_agents()`、`run_store.create_run()`、`run_store.mark_running()`
   - 调用时机：通用 graph 启动后的第一个节点
   - 输入：`RunState`
   - 输出：包含 `persona`、`task`、`record`、`decide_agent`、`validate_agent`、`wait_agent` 的状态增量
   - 状态变化：把运行记录切为 `running`，并把 demo 上下文注入共享状态
   - 异常分支：缺失必要状态时中断 graph，由通用异常兜底补失败报告

4. `backend/graph/run_graph.py::observe_current_page`
   - 职责：在动作前或下一轮开始前读取真实页面状态
   - 调用方：LangGraph
   - 被调用方：`observe_page()`
   - 调用时机：每轮决策前，以及 `log_step` 结束后再次回到主链路时
   - 输入：`session.page`
   - 输出：`current_page_state`
   - 状态变化：刷新当前轮动作前页面事实
   - 异常分支：页面读取失败时中断当前 run

5. `backend/graph/run_graph.py::execute_current_action`
   - 职责：先保存动作前截图并把其路径灌入 `step_before_page_state`，再执行受控动作
   - 调用方：LangGraph
   - 被调用方：`page.screenshot()`、`backend.execution.playwright_adapter.execute_action()`
   - 调用时机：动作决策后
   - 输入：`current_page_state`、`current_action`、`run_id`、`screenshot_dir`
   - 输出：`current_execution_result`、`step_before_page_state`
   - 状态变化：清空上一轮 `post_action_page_state` 与等待观察状态；`ExecutionResult` 只保留动作执行结果，不再承载截图路径和动作后 URL
   - 异常分支：动作执行异常时进入通用异常兜底

6. `backend/graph/run_graph.py::observe_after_action`
   - 职责：对非 wait 动作单独抓取动作后页面状态和截图
   - 调用方：`route_after_execute()` 命中的 `observe_after_action` 分支
   - 被调用方：`observe_page()`、`page.screenshot()`
   - 调用时机：非 wait 动作执行完成后、验证前
   - 输入：`session.page`、`run_id`、`screenshot_dir`
   - 输出：`current_page_state`、`post_action_page_state`
   - 状态变化：把动作后截图路径写入 `post_action_page_state.screenshot_path`
   - 异常分支：页面读取或截图失败时中断当前 run

7. `backend/graph/run_graph.py::wait_after_action`
   - 职责：统一承接普通 `wait` 动作与等待观察分支，并在每轮判断前后保存 trace 截图，结束时再保存动作后页面截图
   - 调用方：普通 `wait` 动作路由，或 `validate_current_progress()` 返回 `recovery_candidate` 后的条件分支
   - 被调用方：`backend.graph.wait_observer.observe_until_ready()`、`wait_agent.ainvoke()`、`page.screenshot()`
   - 调用时机：动作执行后命中普通等待，或验证节点要求继续观察页面时
   - 输入：`session.page`、`task`、`current_action`、`wait_agent`
   - 输出：`post_action_page_state`、`wait_observation_status`、`wait_observation_reason`、`wait_observation_observations`、`wait_observation_elapsed_ms`、`wait_observation_timeout_ms`、`wait_observation_terminal_decision`、`wait_observation_traces`
   - 状态变化：在 `RunState` 中写入等待观察轨迹、累计等待时长、终止决策、逐轮截图路径以及最终动作后页面快照
   - 异常分支：等待观察前置状态缺失时直接失败；超时后会写入 `normal_timeout` 或 `abnormal_stuck` 结论并继续交给验证节点收尾

8. `backend/graph/wait_observer.py::observe_until_ready`
   - 职责：在每轮页面观察与模型判断之间累计真实处理耗时，并为 trace 附带截图路径
   - 调用方：`wait_after_action()`
   - 被调用方：`observe_page()`、`classify_fn()`、可选 `capture_trace_screenshot_fn()`、`page.wait_for_timeout()`
   - 调用时机：进入 wait node 后循环执行
   - 输入：页面观察结果、剩余预算、模型判断结果
   - 输出：`WaitObservationResult`
   - 状态变化：更新 `elapsed_ms`、普通等待预算、异常等待预算与 `WaitObservationTrace.screenshot_path`
   - 异常分支：预算耗尽时返回 `normal_timeout` 或 `abnormal_stuck`，不中断 Python 流程而是交由验证节点决定收尾

9. `backend/graph/run_graph.py::validate_current_progress`
   - 职责：基于动作后页面或等待结束后的页面统一验证任务进展
   - 调用方：LangGraph
   - 被调用方：`validate_agent.ainvoke()`、`backend.analysis.validator.validate_progress()`
   - 调用时机：非 wait 动作执行后，或 `wait_after_action` 结束后
   - 输入：`task`、`current_execution_result`、`current_page_state`、`step_logs`、`wait_observation_terminal_decision`
   - 输出：`ValidationResult`
   - 状态变化：更新 `current_validation_result` 与 `should_stop`
   - 异常分支：缺失验证前置状态时直接失败；wait 终止结论会优先转为固定成功/失败验证结果

10. `backend/graph/run_graph.py::log_current_step`
   - 职责：把动作前页面、动作后页面、执行结果、验证结果和等待观察结果合并写入 `StepLog`
   - 调用方：LangGraph
   - 被调用方：`run_store.add_step()`
   - 调用时机：验证完成后
   - 输入：`step_before_page_state`、`post_action_page_state`、`current_action`、`current_execution_result`、`current_validation_result`
   - 输出：更新后的 `step_logs` 与 `current_step_index`
   - 状态变化：内存步骤列表追加一条包含 before/after 页面状态的 `StepLog`
   - 异常分支：任一必需字段缺失时直接失败

11. `backend/analysis/report_builder.py::_serialize_step`
   - 职责：把 `StepLog` 序列化为模型报告输入时，保留 `before_page_state` 与 `after_page_state`
   - 调用方：`build_run_report()`
   - 被调用方：`ObservedPageState.model_dump()`
   - 调用时机：最终报告生成前
   - 输入：`StepLog`
   - 输出：结构化步骤字典
   - 状态变化：报告分析输入不再拆散页面字段，而是保留完整页面快照对象
   - 异常分支：步骤结构缺字段时由模型序列化前直接报错

### 状态流转

- 成功路径：`queued` → `running` → `succeeded`
- 失败路径：`queued` → `running` → `failed`

### 输出

- `list[StepLog]`
  - 去向：`GET /runs/demo/{run_id}/steps`
  - 用途：查看每一步的动作前页面状态、动作后页面状态、等待观察终止结论，以及逐轮 `wait_observation_traces[].screenshot_path`
- `RunReport`
  - 去向：`GET /runs/demo/{run_id}/report`
  - 用途：查看最终成功失败摘要、摩擦信号、详细 `key_findings`、等待观察摘要，以及每一步 `before_page_state` / `after_page_state` 组成的页面证据链

### 错误分支

- 动作后页面快照缺失
  - 触发条件：`observe_after_action()` 或 `wait_after_action()` 未能写入 `post_action_page_state`
  - 处理方式：`log_current_step()` 直接抛出 `Step log prerequisites are missing.`，由通用异常兜底落失败状态
  - 影响范围：当前 run 无法继续写 step log，最终以失败收尾
- 等待观察逐轮截图缺失
  - 触发条件：未提供 `capture_trace_screenshot_fn()` 或截图阶段失败
  - 处理方式：trace 中对应 `screenshot_path` 为 `null` 或上抛异常；当前实现默认由 `wait_after_action()` 提供截图函数
  - 影响范围：不影响等待判断本身，但会削弱报告证据完整性
- 普通等待超时
  - 触发条件：wait model 在普通等待预算内持续返回 `normal_waiting`，直到达到 `timeout_ms`
  - 处理方式：`observe_until_ready()` 返回 `normal_timeout`，再由验证节点归一化为失败结论
  - 影响范围：当前 step 会保留等待轨迹、逐轮截图与最终动作后页面快照
- 异常卡住超时
  - 触发条件：wait model 持续返回 `abnormal_stuck`，直到达到异常卡住预算上限
  - 处理方式：返回 `abnormal_stuck` 终止决策，再交给验证节点生成失败结论
  - 影响范围：当前 run 可能直接终止为失败，并在步骤日志与报告中留下卡住前后的页面证据
