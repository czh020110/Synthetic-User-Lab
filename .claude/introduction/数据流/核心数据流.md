# 核心数据流

## Demo Run 通过通用 Run Graph 生成带结论的详细报告

### 本次更新原因

- 关联 Commit：区分模型错误并增加重试护栏
- 关联修改记录：`introduction/修改记录/区分模型错误并增加重试护栏.md`
- 更新原因：本次提交为决策/验证 agent 增加了模型 API 错误重试与结构化格式错误重试，要求在模型调用失败时继续保持 agent 驱动而不是回退为无 agent 测试；同时把最终失败状态区分为 `model_error` 与 `system_error`，并让状态接口与最终报告都保留原始错误信息，因此需要同步 run graph、错误兜底和报告输出的数据流说明。

### 链路目标

在保留 `/runs/demo/*` 现有 API 入口的前提下，让 demo 场景只负责提供默认 persona、task 和 demo 起始页，而页面观察、动作决策、动作执行、验证、日志记录、报告生成与异常兜底统一由通用 run graph 承载；当决策或验证 agent 出现模型 API 错误或结构化回复格式错误时，graph 会先在 agent 层按固定上限重试，只有重试耗尽后才以 `model_error` 收尾，并把原始 API 报错或格式校验报错继续透出到 run 状态与最终报告。

### 修改前数据流

`POST /api/v1/runs/demo/start` 仍然通过 demo graph 包装通用 run graph，但决策与验证节点直接调用 agent；agent 返回结构化结果后立即进入 Pydantic 校验，一旦出现 API 异常或格式错误就会在节点内部直接抛错，run 失败时只保留统一的 `error_message`，不会区分模型错误和系统异常。

### 修改后数据流

`POST /api/v1/runs/demo/start` → `backend.api.routes.demo_runs.start_demo_run()` → `backend.graph.demo_run_graph.create_demo_placeholder_record()` → `asyncio.create_task(run_demo_workflow(...))` → `backend.graph.demo_run_graph.run_demo_workflow()` → `backend.graph.run_graph.run_workflow(..., load_demo_context)` → `load_demo_context` → `init_session` → `observe_page` → `decide_action` → `_invoke_agent_with_retries`（API 错误最多重试 5 次，格式错误最多重试 3 次）→ `execute_action` → `validate_progress` → `_invoke_agent_with_retries` → `log_step` → `route_after_log` → `finalize_report` / `run_workflow` 异常兜底 → `backend.stores.in_memory_run_store`

### 输入

- `RunRequest`
  - 来源：用户 / API
  - 关键字段：`run_name`、`headless`
  - 约束：由 `POST /runs/demo/start` 接收；`headless` 为空时回退到系统默认浏览器配置
- `app_base_url`
  - 来源：HTTP 请求
  - 关键字段：当前服务基础地址
  - 约束：由路由层从 `request.base_url` 生成，用于拼接 demo 起始页 `start_url`
- `Persona`
  - 来源：文件 / 代码内默认装配
  - 关键字段：`id`、`name`、`description`、`skill_level`、`patience_level`、`risk_preference`
  - 约束：当前由 `build_demo_persona()` 固定生成，但共享类型已不再限定为 demo 专用命名
- `Task`
  - 来源：文件 / 代码内默认装配
  - 关键字段：`id`、`name`、`description`、`start_url`、`success_criteria`、`max_steps`、`allowed_actions`
  - 约束：当前由 `build_demo_task(app_base_url)` 生成，`start_url` 指向 `/demo/index.html`

### 处理步骤

1. `backend/api/routes/demo_runs.py::start_demo_run`
   - 职责：保留 demo API 入口，创建占位记录并异步启动后台 run
   - 调用方：HTTP 客户端
   - 被调用方：`get_settings()`、`create_demo_placeholder_record()`、`run_store.create_run()`、`run_demo_workflow()`
   - 调用时机：收到 `POST /runs/demo/start` 时
   - 输入：`Request`、`RunRequest`
   - 输出：包含 `run_id` 与 `queued` 状态的 `ApiResponse`
   - 状态变化：在内存存储中创建 `RunRecord(status=queued)`
   - 异常分支：创建后续流程失败时由后台 run 负责落失败状态；接口自身仅处理同步阶段异常

2. `backend/graph/demo_run_graph.py::create_demo_placeholder_record`
   - 职责：把 demo 默认 persona/task 组装到占位运行记录
   - 调用方：`start_demo_run()`
   - 被调用方：`build_demo_persona()`、`build_demo_task()`
   - 调用时机：路由层创建占位记录时
   - 输入：`run_id`、`RunRequest`、`app_base_url`
   - 输出：`RunRecord`
   - 状态变化：无额外副作用，仅返回待写入存储的记录
   - 异常分支：demo 默认上下文缺失时抛出异常，接口启动失败

3. `backend/graph/demo_run_graph.py::load_demo_context`
   - 职责：在真正执行前加载 demo 场景的默认 `Persona`、`Task`、agent 与运行记录
   - 调用方：`backend.graph.run_graph.build_run_graph()` 生成的 LangGraph
   - 被调用方：`build_demo_persona()`、`build_demo_task()`、`create_run_agents()`、`run_store.create_run()`、`run_store.mark_running()`
   - 调用时机：通用 graph 启动后的第一个节点
   - 输入：`RunState`
   - 输出：包含 `persona`、`task`、`record`、`decide_agent`、`validate_agent` 的状态增量
   - 状态变化：把运行记录切为 `running`，并把 demo 上下文注入共享状态
   - 异常分支：缺失必要状态时中断 graph，由通用异常兜底补失败报告

4. `backend/graph/run_graph.py::init_session`
   - 职责：依据共享 `Task.start_url` 初始化浏览器并进入起始页面
   - 调用方：LangGraph
   - 被调用方：`create_browser_session()`
   - 调用时机：`load_context` 后
   - 输入：`RunState.request`、`RunState.task`
   - 输出：`session`
   - 状态变化：在 graph state 中写入浏览器会话
   - 异常分支：`task` 缺失或浏览器启动失败时抛出异常，由通用失败兜底处理

5. `backend/graph/run_graph.py::observe_current_page`
   - 职责：读取真实页面状态，供决策和验证使用
   - 调用方：LangGraph
   - 被调用方：`observe_page()`
   - 调用时机：每轮决策前
   - 输入：`session.page`
   - 输出：`ObservedPageState`
   - 状态变化：更新 `current_page_state`
   - 异常分支：页面读取失败时中断当前 run

6. `backend/graph/run_graph.py::decide_next_action`
   - 职责：基于共享 `Persona`、`Task`、当前页面事实与最近步骤生成受控动作；若 agent 先返回结构化格式错误，会在同一节点追加格式修正 prompt 重试，若连续 API 调用失败则最终抛出 `model_error`。
   - 调用方：LangGraph
   - 被调用方：`backend.prompt.graph.decide_input.format_messages()`、`decide_agent.ainvoke()`、`backend.prompt.graph.format_retry_input`
   - 调用时机：页面观察后
   - 输入：`current_page_state`、`step_logs`、`decide_agent`
   - 输出：`ActionInput`
   - 状态变化：更新 `current_action`
   - 异常分支：页面状态、task 或 agent 缺失时直接失败；API 错误重试耗尽后抛出 `ModelInvocationError(stage='decide')`，格式错误重试耗尽后同样转为 `model_error`

7. `backend/graph/run_graph.py::execute_current_action`
   - 职责：执行受控动作并生成截图
   - 调用方：LangGraph
   - 被调用方：`execute_action()`
   - 调用时机：动作决策后
   - 输入：`session`、`current_action`、`run_id`、`screenshot_dir`
   - 输出：`ExecutionResult`
   - 状态变化：更新 `current_execution_result`
   - 异常分支：动作执行异常时进入通用异常兜底

8. `backend/graph/run_graph.py::validate_current_progress`
   - 职责：先用 `validate_agent` 生成结构化验证，再交给规则护栏统一补强失败、卡住、偏航边界；若 agent 返回格式错误，会追加格式修正 prompt 重试，若 API 连续失败则最终抛出 `model_error`。
   - 调用方：LangGraph
   - 被调用方：`observe_page()`、`validate_agent.ainvoke()`、`backend.analysis.validator.validate_progress()`、`backend.prompt.graph.format_retry_input`
   - 调用时机：动作执行后
   - 输入：`task`、`current_execution_result`、`step_logs`、`current_action`
   - 输出：`ValidationResult`
   - 状态变化：更新 `current_page_state`、`current_validation_result`、`should_stop`
   - 异常分支：缺失验证前置状态时直接失败；非成功场景由规则护栏继续兜底失败、卡住和偏航边界；API 或格式重试耗尽后转为 `ModelInvocationError(stage='validate')`

9. `backend/graph/run_graph.py::log_current_step`
   - 职责：把本轮页面状态、动作、执行结果和验证结果写入步骤日志
   - 调用方：LangGraph
   - 被调用方：`run_store.add_step()`
   - 调用时机：验证完成后
   - 输入：当前轮状态字段
   - 输出：更新后的 `step_logs` 与 `current_step_index`
   - 状态变化：内存步骤列表追加一条 `StepLog`
   - 异常分支：任一必需字段缺失时直接失败

10. `backend/graph/run_graph.py::finalize_report`
   - 职责：生成最终报告、写回存储并关闭浏览器；报告需要输出统一的 `conclusion` 结论级别，并保留 `error_type` / `error_message`，让 `key_findings`、`next_recommendations` 只来自本次 run 的真实步骤、摩擦信号、失败原因与 persona 行为特征；报告分析 prompt 由 `backend.prompt.report.RECOMMENDATION_SYSTEM_PROMPT` 提供，且其文案明确说明 `keep` / `optimize` / `fix` 的含义。
   - 调用方：LangGraph
   - 被调用方：`build_run_report()`、`run_store.complete_run()`、`close_browser_session()`
   - 调用时机：`should_stop=True` 后
   - 输入：`record`、`step_logs`、`session`
   - 输出：`RunReport`
   - 状态变化：写入最终 `RunReport`，同步更新 `RunRecord.status`
   - 异常分支：即使报告阶段出错也会在 `finally` 中关闭浏览器

11. `backend/graph/run_graph.py::run_workflow`
   - 职责：统一承载通用 graph 编译、执行和异常兜底补失败报告；当异常属于模型调用问题时，失败状态会标记为 `model_error` 并原样保留 API 错误信息，否则标记为 `system_error`。
   - 调用方：`backend.graph.demo_run_graph.run_demo_workflow()`
   - 被调用方：`build_run_graph()`、`graph.ainvoke()`、`run_store.fail_run()`、`run_store.complete_run()`
   - 调用时机：后台任务实际执行时
   - 输入：`run_id`、`RunRequest`、`app_base_url`、`screenshot_dir`、`load_context_node`
   - 输出：最终 graph state
   - 状态变化：异常时把 run 标记为 `failed`，并在已有记录但缺少 report 时补最小失败报告；补报告时固定写入 `conclusion="fix"`，并同步 `error_type` / `error_message`
   - 异常分支：统一封装异常类型与信息，避免 `/report` 长时间无返回

### 状态流转

- 成功路径：`queued` → `running` → `succeeded`
- 失败路径：`queued` → `running` → `failed`

### 输出

- `RunStatusResponse`
  - 去向：`GET /runs/demo/{run_id}`
  - 用途：轮询当前 run 状态、`error_type` 与原始 `error_message`；当模型重试耗尽时明确返回 `model_error`
- `list[StepLog]`
  - 去向：`GET /runs/demo/{run_id}/steps`
  - 用途：查看每一步页面观察、动作、执行结果和验证结论
- `RunReport`
  - 去向：`GET /runs/demo/{run_id}/report`
  - 用途：查看最终成功失败摘要、摩擦信号、详细 `key_findings`、`conclusion`、`error_type`、`error_message` 与后续建议；`conclusion` 只允许为 `keep` / `optimize` / `fix`，且建议只对应本次 run 中真实出现的问题与 persona 行为特征，不写项目路线图式规划

### 错误分支

- 模型调用错误或格式重试耗尽
  - 触发条件：`decide_agent` / `validate_agent` 连续 API 调用失败超过 5 次，或结构化回复格式连续校验失败超过 3 次
  - 处理方式：`_invoke_agent_with_retries()` 抛出 `ModelInvocationError`，`run_workflow()` 调用 `run_store.fail_run(..., error_type="model_error")`，并在必要时补带有原始错误信息的失败报告
  - 影响范围：当前 run 结束为 `failed`，状态接口和最终报告都会暴露 `error_type=model_error`
- 浏览器初始化失败或 graph 其他节点抛异常
  - 触发条件：`task` 缺失、Playwright 启动失败、非模型调用类节点内部异常
  - 处理方式：`run_workflow()` 调用 `run_store.fail_run(..., error_type="system_error")`，必要时基于已有步骤补失败报告
  - 影响范围：当前 run 结束为 `failed`，但不会影响其他 run
- 动作执行失败
  - 触发条件：`execute_action()` 返回 `success=False`
  - 处理方式：规则护栏输出 `failed + action_failed`
  - 影响范围：当前 step 终止并进入最终报告
- agent 给出成功结论
  - 触发条件：`validate_agent` 基于 `task.description`、`success_criteria`、最新页面状态与执行结果判断任务已经完成
  - 处理方式：`backend.analysis.validator.validate_progress()` 将该结论归一化为 `succeeded + should_stop=true`，随后进入最终报告生成
  - 影响范围：当前 run 直接结束为成功，后续不再继续执行额外动作

### 相关文件

- `backend/api/routes/demo_runs.py`：demo API 入口，只保留启动和查询职责
- `backend/graph/demo_run_graph.py`：demo 默认 persona/task 与通用 graph 包装层
- `backend/graph/run_graph.py`：通用运行节点、agent 重试护栏、图编排和异常兜底入口
- `backend/prompt/graph.py`：graph 决策/验证提示词与格式错误重试提示词的统一来源
- `backend/prompt/report.py`：报告分析 system prompt 的统一来源，并定义 `conclusion` 三档取值的中文含义说明
- `backend/prompt/__init__.py`：统一导出 prompt 包公共入口
- `backend/graph/run_state.py`：通用 `RunState` 状态定义
- `backend/schemas/run_schemas.py`：通用 `Persona`、`Task`、`RunRecord`、`RunStatusResponse`、`RunReport` 等共享契约，新增 `RunErrorType`、`error_type` 与 `error_message`
- `backend/analysis/validator.py`：规则护栏验证入口
- `backend/analysis/report_builder.py`：基于主记录与步骤日志生成最终 run 报告，并区分模型调用错误与系统异常的报告文案
- `backend/stores/in_memory_run_store.py`：运行状态、步骤和报告的内存存储，失败状态会保留 `error_type`
- `tests/test_validator.py`：验证 agent API 错误重试、格式错误重试、超限失败与 `model_error` 状态/报告回归
- `tests/test_report_builder.py`：验证报告结论级别、详细报告内容与错误字段透出边界
- `tests/test_demo_run_api.py`：验证 demo API 入口与最终报告结构在模型错误处理升级后未回归
