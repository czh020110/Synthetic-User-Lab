# 核心数据流

## Demo Run 通过通用 Run Graph 统一普通 wait 与等待观察链路

### 本次更新原因

- 关联 Commit：统一等待观察链路以支撑模型化动态等待
- 关联修改记录：`introduction/修改记录/统一等待观察链路以支撑模型化动态等待.md`
- 更新原因：本次提交把普通 `wait` 动作与等待观察节点收敛到同一条 wait node 链路，由 wait model 在每轮观察中决定等待类型与 `next_wait_ms`；同时把累计等待时长、超时预算、终止决策与 wait trace 写入步骤日志和最终报告，因此需要同步 run graph、wait observer、状态结构、报告输出和环境配置的数据流说明。

### 链路目标

在保留 `/runs/demo/*` 现有 API 入口的前提下，让 demo 场景只负责提供默认 persona、task 和 demo 起始页，而页面观察、动作决策、动作执行、动态等待、结果验证、日志记录、报告生成与异常兜底统一由通用 run graph 承载；当 agent 选择普通 `wait` 动作，或验证结果命中 `recovery_candidate` 需要继续观察页面时，graph 都会进入同一个 `wait_after_action` 节点，由快速模型决定本轮继续等待多久、页面是否已恢复可继续操作，或任务是否已经完成。

### 修改前数据流

`POST /api/v1/runs/demo/start` 通过 demo graph 包装通用 run graph。只有验证节点命中 `stuck_page + recovery_candidate` 时才会进入等待观察分支；普通 `wait` 动作会在 Playwright 执行器里直接调用 `page.wait_for_timeout(...)`，既不会留下统一的 wait trace，也不能让模型动态决定下一轮等待时长和等待类型。

### 修改后数据流

`POST /api/v1/runs/demo/start` → `backend.api.routes.demo_runs.start_demo_run()` → `backend.graph.demo_run_graph.create_demo_placeholder_record()` → `asyncio.create_task(run_demo_workflow(...))` → `backend.graph.demo_run_graph.run_demo_workflow()` → `backend.graph.run_graph.run_workflow(..., load_demo_context)` → `load_demo_context` → `init_session` → `observe_page` → `decide_action` → `execute_action` → `wait_after_action` / `observe_after_action` → `validate_progress` / `log_step` → `route_after_log` → `finalize_report` / `run_workflow` 异常兜底 → `backend.stores.in_memory_run_store`

### 输入

- `RunRequest`
  - 来源：用户 / API
  - 关键字段：`run_name`、`headless`
  - 约束：由 `POST /runs/demo/start` 接收；`headless` 为空时回退到系统默认浏览器配置
- `app_base_url`
  - 来源：HTTP 请求
  - 关键字段：当前服务基础地址
  - 约束：由路由层从 `request.base_url` 生成，用于拼接 demo 起始页 `start_url`
- `Persona`
  - 来源：文件 / 代码内默认装配
  - 关键字段：`id`、`name`、`description`、`skill_level`、`patience_level`、`risk_preference`
  - 约束：当前由 `build_demo_persona()` 固定生成，但共享类型已不再限定为 demo 专用命名
- `Task`
  - 来源：文件 / 代码内默认装配
  - 关键字段：`id`、`name`、`description`、`start_url`、`success_criteria`、`max_steps`、`allowed_actions`
  - 约束：当前由 `build_demo_task(app_base_url)` 生成，`start_url` 指向 `/demo/index.html`，任务描述不直接泄露固定等待时长
- `WaitObservationOptions`
  - 来源：代码默认配置
  - 关键字段：`interval_ms`、`timeout_ms`
  - 约束：普通等待默认受 `normal_waiting` 预算约束，异常卡住默认受 `abnormal_stuck` 预算约束；wait model 每轮返回的 `next_wait_ms` 不能突破当前预算

### 处理步骤

1. `backend/api/routes/demo_runs.py::start_demo_run`
   - 职责：保留 demo API 入口，创建占位记录并异步启动后台 run
   - 调用方：HTTP 客户端
   - 被调用方：`get_settings()`、`create_demo_placeholder_record()`、`run_store.create_run()`、`run_demo_workflow()`
   - 调用时机：收到 `POST /runs/demo/start` 时
   - 输入：`Request`、`RunRequest`
   - 输出：包含 `run_id` 与 `queued` 状态的 `ApiResponse`
   - 状态变化：在内存存储中创建 `RunRecord(status=queued)`
   - 异常分支：创建后续流程失败时由后台 run 负责落失败状态；接口自身仅处理同步阶段异常

2. `backend/graph/demo_run_graph.py::create_demo_placeholder_record`
   - 职责：把 demo 默认 persona/task 组装到占位运行记录
   - 调用方：`start_demo_run()`
   - 被调用方：`build_demo_persona()`、`build_demo_task()`
   - 调用时机：路由层创建占位记录时
   - 输入：`run_id`、`RunRequest`、`app_base_url`
   - 输出：`RunRecord`
   - 状态变化：无额外副作用，仅返回待写入存储的记录
   - 异常分支：demo 默认上下文缺失时抛出异常，接口启动失败

3. `backend/graph/demo_run_graph.py::load_demo_context`
   - 职责：在真正执行前加载 demo 场景的默认 `Persona`、`Task`、agent 与运行记录
   - 调用方：`backend.graph.run_graph.build_run_graph()` 生成的 LangGraph
   - 被调用方：`build_demo_persona()`、`build_demo_task()`、`create_run_agents()`、`run_store.create_run()`、`run_store.mark_running()`
   - 调用时机：通用 graph 启动后的第一个节点
   - 输入：`RunState`
   - 输出：包含 `persona`、`task`、`record`、`decide_agent`、`validate_agent`、`wait_agent` 的状态增量
   - 状态变化：把运行记录切为 `running`，并把 demo 上下文注入共享状态
   - 异常分支：缺失必要状态时中断 graph，由通用异常兜底补失败报告

4. `backend/graph/run_graph.py::init_session`
   - 职责：依据共享 `Task.start_url` 初始化浏览器并进入起始页面
   - 调用方：LangGraph
   - 被调用方：`create_browser_session()`
   - 调用时机：`load_context` 后
   - 输入：`RunState.request`、`RunState.task`
   - 输出：`session`
   - 状态变化：在 graph state 中写入浏览器会话
   - 异常分支：`task` 缺失或浏览器启动失败时抛出异常，由通用失败兜底处理

5. `backend/graph/run_graph.py::observe_current_page`
   - 职责：读取真实页面状态，供决策和验证使用
   - 调用方：LangGraph
   - 被调用方：`observe_page()`
   - 调用时机：每轮决策前，以及等待结束后需要刷新页面事实时
   - 输入：`session.page`
   - 输出：`ObservedPageState`
   - 状态变化：更新 `current_page_state`
   - 异常分支：页面读取失败时中断当前 run

6. `backend/graph/run_graph.py::decide_next_action`
   - 职责：基于共享 `Persona`、`Task`、当前页面事实与最近步骤生成受控动作；若 agent 先返回结构化格式错误，会在同一节点追加格式修正 prompt 重试，若连续 API 调用失败则最终抛出 `model_error`
   - 调用方：LangGraph
   - 被调用方：`backend.prompt.graph.decide_input.format_messages()`、`decide_agent.ainvoke()`、`backend.prompt.graph.format_retry_input`
   - 调用时机：页面观察后
   - 输入：`current_page_state`、`step_logs`、`decide_agent`
   - 输出：`ActionInput`
   - 状态变化：更新 `current_action`
   - 异常分支：页面状态、task 或 agent 缺失时直接失败；API 错误重试耗尽后抛出 `ModelInvocationError(stage='decide')`，格式错误重试耗尽后同样转为 `model_error`

7. `backend/graph/run_graph.py::execute_current_action`
   - 职责：执行受控动作并生成截图；当动作类型为 `wait` 时只记录本次等待请求，不再直接调用 `page.wait_for_timeout(...)`
   - 调用方：LangGraph
   - 被调用方：`execute_action()`
   - 调用时机：动作决策后
   - 输入：`session`、`current_action`、`run_id`、`screenshot_dir`
   - 输出：`ExecutionResult`
   - 状态变化：更新 `current_execution_result`，并清空上一轮等待观察状态
   - 异常分支：动作执行异常时进入通用异常兜底

8. `backend/graph/run_graph.py::wait_after_action`
   - 职责：统一承接普通 `wait` 动作与等待观察分支，使用快速模型循环判断页面处于 `normal_waiting`、`abnormal_stuck`、`ready_for_next_action` 还是 `task_completed`，并决定 `next_wait_ms`
   - 调用方：普通 `wait` 动作路由，或 `validate_current_progress()` 返回 `recovery_candidate` 后的条件分支
   - 被调用方：`backend.graph.wait_observer.observe_until_ready()`、`wait_agent.ainvoke()`、`backend.prompt.graph.wait_observe_input`
   - 调用时机：动作执行后命中普通等待，或验证节点要求继续观察页面时
   - 输入：`session.page`、`task`、`current_action`、`wait_agent`
   - 输出：`wait_observation_status`、`wait_observation_reason`、`wait_observation_observations`、`wait_observation_elapsed_ms`、`wait_observation_timeout_ms`、`wait_observation_terminal_decision`、`wait_observation_traces`
   - 状态变化：在 `RunState` 中写入等待观察轨迹、累计等待时长、当前预算和终止决策，并在必要时覆盖 `current_page_state`
   - 异常分支：等待观察前置状态缺失时直接失败；超时后会写入 `normal_timeout` 或 `abnormal_stuck` 结论并继续交给验证节点收尾

9. `backend/graph/run_graph.py::validate_current_progress`
   - 职责：在动作后观察或等待结束后统一验证任务进展；验证结果会结合 `current_action`、等待终止决策和规则护栏，处理 `success`、`normal_timeout`、`abnormal_stuck` 等结论
   - 调用方：LangGraph
   - 被调用方：`observe_page()`、`validate_agent.ainvoke()`、`backend.analysis.validator.validate_progress()`、`backend.prompt.graph.format_retry_input`
   - 调用时机：非 wait 动作执行后，或 `wait_after_action` 结束后
   - 输入：`task`、`current_execution_result`、`step_logs`、`current_action`、`wait_observation_terminal_decision`
   - 输出：`ValidationResult`
   - 状态变化：更新 `current_page_state`、`current_validation_result`、`should_stop`
   - 异常分支：缺失验证前置状态时直接失败；非成功场景由规则护栏继续兜底失败、卡住和偏航边界；API 或格式重试耗尽后转为 `ModelInvocationError(stage='validate')`

10. `backend/graph/run_graph.py::log_current_step`
   - 职责：把本轮页面状态、动作、执行结果、验证结果和等待观察结果写入步骤日志
   - 调用方：LangGraph
   - 被调用方：`run_store.add_step()`
   - 调用时机：验证完成后
   - 输入：当前轮状态字段
   - 输出：更新后的 `step_logs` 与 `current_step_index`
   - 状态变化：内存步骤列表追加一条 `StepLog`
   - 异常分支：任一必需字段缺失时直接失败

11. `backend/graph/run_graph.py::finalize_report`
   - 职责：生成最终报告、写回存储并关闭浏览器；报告需要输出统一的 `conclusion` 结论级别，并保留 `error_type` / `error_message` 以及等待观察摘要，让 `key_findings`、`next_recommendations` 只来自本次 run 的真实步骤、摩擦信号、失败原因、等待轨迹与 persona 行为特征
   - 调用方：LangGraph
   - 被调用方：`build_run_report()`、`run_store.complete_run()`、`close_browser_session()`
   - 调用时机：`should_stop=True` 后
   - 输入：`record`、`step_logs`、`session`
   - 输出：`RunReport`
   - 状态变化：写入最终 `RunReport`，同步更新 `RunRecord.status`
   - 异常分支：即使报告阶段出错也会在 `finally` 中关闭浏览器

12. `backend/graph/run_graph.py::run_workflow`
   - 职责：统一承载通用 graph 编译、执行和异常兜底补失败报告；当异常属于模型调用问题时，失败状态会标记为 `model_error` 并原样保留 API 错误信息，否则标记为 `system_error`，同时兜底关闭浏览器会话避免资源泄漏
   - 调用方：`backend.graph.demo_run_graph.run_demo_workflow()`
   - 被调用方：`build_run_graph()`、`graph.ainvoke()`、`run_store.fail_run()`、`run_store.complete_run()`、`close_browser_session()`
   - 调用时机：后台任务实际执行时
   - 输入：`run_id`、`RunRequest`、`app_base_url`、`screenshot_dir`、`load_context_node`
   - 输出：最终 graph state
   - 状态变化：异常时把 run 标记为 `failed`，并在已有记录但缺少 report 时补最小失败报告；补报告时固定写入 `conclusion="fix"`，并同步 `error_type` / `error_message`
   - 异常分支：统一封装异常类型与信息，避免 `/report` 长时间无返回

### 状态流转

- 成功路径：`queued` → `running` → `succeeded`
- 失败路径：`queued` → `running` → `failed`

### 输出

- `RunStatusResponse`
  - 去向：`GET /runs/demo/{run_id}`
  - 用途：轮询当前 run 状态、`error_type` 与原始 `error_message`；当模型重试耗尽时明确返回 `model_error`
- `list[StepLog]`
  - 去向：`GET /runs/demo/{run_id}/steps`
  - 用途：查看每一步页面观察、动作、执行结果、验证结论，以及 `wait_observation_status/reason/elapsed_ms/timeout_ms/terminal_decision/traces`
- `RunReport`
  - 去向：`GET /runs/demo/{run_id}/report`
  - 用途：查看最终成功失败摘要、摩擦信号、详细 `key_findings`、`conclusion`、`error_type`、`error_message`、等待观察摘要与后续建议；`conclusion` 只允许为 `keep` / `optimize` / `fix`，且建议只对应本次 run 中真实出现的问题与 persona 行为特征，不写项目路线图式规划

### 错误分支

- 模型调用错误或格式重试耗尽
  - 触发条件：`decide_agent` / `validate_agent` / `wait_agent` 连续 API 调用失败超过 5 次，或结构化回复格式连续校验失败超过 3 次
  - 处理方式：`_invoke_agent_with_retries()` 抛出 `ModelInvocationError`，`run_workflow()` 调用 `run_store.fail_run(..., error_type="model_error")`，并在必要时补带有原始错误信息的失败报告
  - 影响范围：当前 run 结束为 `failed`，状态接口和最终报告都会暴露 `error_type=model_error`
- 浏览器初始化失败或 graph 其他节点抛异常
  - 触发条件：`task` 缺失、Playwright 启动失败、非模型调用类节点内部异常
  - 处理方式：`run_workflow()` 调用 `run_store.fail_run(..., error_type="system_error")`，必要时基于已有步骤补失败报告，并兜底关闭浏览器
  - 影响范围：当前 run 结束为 `failed`，但不会影响其他 run
- 普通等待超时
  - 触发条件：wait model 在普通等待预算内持续返回 `normal_waiting`，直到达到 `timeout_ms`
  - 处理方式：`observe_until_ready()` 返回 `normal_timeout` 结论，`validate_current_progress()` 结合页面最新状态决定继续、失败或收尾
  - 影响范围：当前 step 会保留完整等待轨迹，并在步骤日志和报告中暴露累计等待结果
- 异常卡住超时
  - 触发条件：wait model 持续返回 `abnormal_stuck`，直到达到异常卡住预算上限
  - 处理方式：等待预算不会在 `normal_waiting` 后被重置，超时后写入 `abnormal_stuck` 终止决策，再交给验证护栏决定失败收尾
  - 影响范围：当前 run 可能直接终止为失败，并在报告中留下卡住证据
- 页面恢复可继续
  - 触发条件：wait model 返回 `ready_for_next_action`
  - 处理方式：停止继续等待，记录 actionable 结论和 wait trace，再回到验证节点判断是否继续主链路
  - 影响范围：当前 run 不会继续盲等，但会把本次等待过程保留下来供报告消费
- 任务在等待中完成
  - 触发条件：wait model 返回 `task_completed`
  - 处理方式：停止等待并将终止决策带入 `validate_current_progress()`，由验证节点归一化为 `succeeded + should_stop=true`
  - 影响范围：当前 run 可直接结束为成功，且报告保留等待完成证据

### 相关文件

- `backend/api/routes/demo_runs.py`：demo API 入口，只保留启动和查询职责
- `backend/graph/demo_run_graph.py`：demo 默认 persona/task 与通用 graph 包装层
- `backend/graph/run_graph.py`：通用运行节点、wait 路由、验证收尾与异常兜底入口
- `backend/graph/wait_observer.py`：模型化等待观察循环、等待类型判定、动态 `next_wait_ms` 与轨迹结构
- `backend/execution/playwright_adapter.py`：受控动作执行器，`wait` 动作不再直接在执行器内完成真实等待
- `backend/prompt/graph.py`：graph 决策/验证/等待观察提示词与格式错误重试提示词的统一来源
- `backend/graph/run_state.py`：通用 `RunState` 状态定义，承载等待累计时长、终止决策与 trace 字段
- `backend/schemas/run_schemas.py`：通用 `Persona`、`Task`、`RunRecord`、`RunStatusResponse`、`RunReport` 与 `StepLog` 共享契约，新增等待观察明细字段
- `backend/analysis/report_builder.py`：基于主记录与步骤日志生成最终 run 报告，并序列化等待观察摘要
- `tests/test_validator.py`：验证等待结束后的验证收尾、`success` / `normal_timeout` / `abnormal_stuck` 等分支回归
- `tests/test_wait_observer.py`：验证模型化等待循环返回 `normal_waiting`、`abnormal_stuck`、`ready_for_next_action`、`task_completed` 与对应预算行为
