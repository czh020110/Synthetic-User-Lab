# 目标边界

## 总体目标

构建一个面向 Web 与 AI 产品的 Synthetic User Lab，在受控动作边界内完成 persona 驱动的任务执行、页面观察、进展验证、步骤记录与报告生成，并为后续接入 RAG、持久化与前端查看页保留清晰扩展位。

## 当前阶段目标

- [ ] 固化最小 Demo Run 闭环：验收方式：可通过 FastAPI 启动固定 demo run，并查询 status、steps、report，且 LangGraph 能完成 observe → decide → execute → validate → log → finalize 的闭环。
- [ ] 稳定 run 级 agent 决策、验证与等待观察链路：验收方式：`demo_run_graph` 中 decide / validate / wait observe agent 仅在 run 初始化时创建一次，并按 `run_id` 隔离消息上下文；decide_agent 只输出受控 `ActionInput(action, payload, reason)`，validate_agent 之后再通过规则护栏与等待观察节点稳定收敛重复等待、重复动作、卡住和偏航边界，并透传 retrieval_context。
- [ ] 保留最小恢复候选与收尾路径：验收方式：首次命中 `recovery_candidate` 时，可进入等待观察分支继续判断页面是否可继续，并把等待结果写入步骤日志与最终报告；超时后仍能生成可读失败报告。
- [ ] 保持本地可复现的开发与验证路径：验收方式：`.env`、推荐端口、Conda 环境、测试入口和核心文档保持一致，接手开发者可按文档完成最小验证。

## 成功标准

- [ ] `POST /runs/demo/start` 可返回排队中的 `run_id`，并由异步工作流继续处理。
- [ ] 固定 demo 页面可被 Playwright 会话打开、观察、执行受控动作，并产生步骤日志与截图路径；受控动作统一使用按动作区分的 `payload`，不再依赖旧的 `target/value` 平面字段。
- [ ] run 完成后可从内存存储查询状态、步骤日志和最终报告；异常失败时也能返回可读失败原因与失败报告，且最终报告摘要直接反映最后一步的 `progress_summary`。
- [ ] 最终报告输出与异常补报告路径保持同一结构：`RunReport` 必须包含 `conclusion` 字段，且值只允许为 `keep` / `optimize` / `fix`。
- [ ] 当前 prompt 职责明确区分固定 persona/task 上下文、每步动态页面上下文与 run 级 retrieval_context。
- [ ] 当验证结果出现 `stuck_page + recovery_candidate` 时，run graph 可进入等待观察节点，并把 `wait_observation_status`、`wait_observation_reason`、`wait_observation_observations` 与 `wait_observation_traces` 记录到 `StepLog`。

## 非目标

- 不是当前阶段目标的内容：projects / personas / tasks 正式 CRUD。
- 不是当前阶段目标的内容：PostgreSQL、Redis、对象存储等正式持久化接入。
- 不是当前阶段目标的内容：RAG 检索、失败恢复策略库、多轮恢复动作编排与版本对比的完整实现。
- 不是当前阶段目标的内容：MVP 前端查看页、并发 run、移动端或桌面端扩展。

## 质量底线

- 测试底线：涉及 demo run 主链路、validator、wait observer 或状态查询的修改，至少保留可执行的最小测试或人工验证路径。
- 安全底线：只允许受控 action schema，不允许直接执行任意代码，不默认开放支付、删除、发布等高风险动作。
- 性能底线：当前暂无严格性能指标，但最小 demo run 不应引入明显阻塞式重复初始化和跨 run 上下文污染；等待观察节点应优先复用快速模型配置。
- 文档底线：项目目标、目标边界、TODO、核心数据流与对应 git commit 的详细描述必须与当前已落地实现一致，不记录模板化假设。
- 报告底线：最终运行报告中的 `key_findings`、`next_recommendations` 与 `conclusion` 必须基于本次 run 的真实执行过程、摩擦信号、失败原因、等待观察结果与 persona 行为特征生成；其中 `conclusion` 只允许使用 `keep` / `optimize` / `fix`，不要写回旧值，也不要写入项目路线图、技术债方向或开发规划建议。
- persona 底线：模型需要尽量按当前 persona 的行为特征去模拟与复现可能导致的问题，并在报告中指出该 persona 对应用户群体为什么会在这些步骤失败、犹豫或误操作；如果本次运行没有明显问题，就不要强行生成建议。

## 推进节奏

- 每次迭代优先完成一个可验收的小模块任务，再同步 TODO、数据流与 git commit 的详细描述。
- 在最小闭环稳定前，优先补齐验证、停止条件、等待观察和失败链路，不提前扩展无验证价值的大而全能力。
- 新能力优先沿现有 demo run 主链路增量接入，避免一次性推翻当前闭环。
