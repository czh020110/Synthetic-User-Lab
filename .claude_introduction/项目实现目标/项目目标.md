# 项目目标

## 总体流程目标

以固定 demo persona、固定 demo task 和内置 demo 页面为起点，先跑通从 FastAPI 启动 run、LangGraph 编排执行、Playwright 观察与交互、agent 决策与验证、步骤日志沉淀到最终报告输出的最小纵向闭环；在此基础上逐步扩展为支持多 persona、多任务模板、基础 RAG 上下文、失败恢复、摩擦分析和正式持久化的 Synthetic User Lab。

## 核心功能目标

- [ ] 提供可启动和查询的 demo run API：验收方式：可通过 `/runs/demo/start`、`/runs/demo/{id}`、`/runs/demo/{id}/steps`、`/runs/demo/{id}/report` 完成一次 run 生命周期验证。
- [ ] 提供受控动作执行与页面观察能力：验收方式：Playwright 能在 demo 页面完成 navigate、click、fill、wait、press、scroll、upload、select、hover、check、uncheck、dblclick、drag、ask_for_help、abandon 等动作，动作参数统一由分类型 `payload` 承载，并生成 `ObservedPageState` 与 `ExecutionResult`；终止动作（ask_for_help、abandon）执行后直接进入验证并终止 run。
- [ ] 提供 run 级 agent 决策与验证能力：验收方式：`decide_agent` 与 `validate_agent` 可基于 persona、task、页面事实和历史步骤输出结构化 `ActionInput` 与 `ValidationResult`，并由规则护栏稳定收敛成功、失败、卡住、偏航和等待观察边界。
- [ ] 提供等待观察与最小恢复支撑：验收方式：当验证结果命中 `recovery_candidate` 时，run graph 能进入 `wait_after_action` / `observe_until_ready` 链路，并把等待判断结果写入 `StepLog` 与最终报告。
- [ ] 提供最小状态沉淀与收尾能力：验收方式：内存存储可记录 `RunRecord`、`StepLog`、`RunReport`，异常失败时仍能返回失败报告，且最终报告中的 `conclusion` 统一使用 `keep` / `optimize` / `fix`。
- [ ] 提供基础 RAG 上下文供给能力：验收方式：决策与验证节点能接收产品知识与失败案例检索上下文，并将其写入步骤日志与报告明细。

## 阶段功能范围

- 最小 Demo Run 闭环：目标：打通固定 demo persona、固定 demo task、固定 demo 页面的一次 run 执行、日志与报告闭环；状态：完成；依赖：无。
- Agent 化决策与验证：目标：将 decide 与 validate 两个语义节点都接入 run 级 agent，并区分固定 prompt、动态输入职责和受控动作 payload schema；状态：完成；依赖：最小 Demo Run 闭环。
- 失败信息与本地配置补强：目标：补齐 `.env` 配置加载、API 前缀规范化、失败原因保留和异常报告兜底；状态：完成；依赖：最小 Demo Run 闭环。
- 停止条件与任务验证增强：目标：根据成功条件、步数、重复动作、卡住信号、偏航信号和等待观察结果稳定判断继续、成功、失败或恢复候选；状态：完成；依赖：Agent 化决策与验证。
- 等待观察与最小恢复分支：目标：当页面进入稳定等待态时，通过等待观察节点判断是否可继续，给后续受控恢复动作预留统一入口；状态：进行中；依赖：停止条件与任务验证增强。
- 运行报告结构增强：目标：让最终 `RunReport` 输出统一的 `conclusion` 结论级别、基于步骤证据的详细 `key_findings`，并让建议只对应本次 run 的真实问题与 persona 行为特征；状态：完成；依赖：停止条件与任务验证增强。
- 基础 RAG 上下文层最小闭环：目标：在决策、验证、步骤日志和报告中透出产品知识与失败案例检索上下文；状态：完成；依赖：停止条件与任务验证增强。
- 持久化、UI 经验库与摩擦分析扩展：目标：补齐正式存储、UI 经验库、失败恢复策略库与结构化摩擦分析；状态：进行中；依赖：基础 RAG 上下文层最小闭环。

## 实现发现

- 当前仓库已不只是纯规则式 demo run，`demo_run_graph` 中决策与验证节点都已改为 agent 结构化输出，但状态存储仍为 `InMemoryRunStore`。
- 当前 task 成功判定已从单一 `success_text` 收敛为 `success_criteria` 列表，启动请求与任务定义的职责边界更清晰。
- 每步 `validate_agent` 输入现在会显式携带 `task.success_criteria`，避免模型只依据页面文案和最近步骤臆测完成条件；同时已补齐 fill 同页、软失败继续观察、连续信号重置等停止条件边界回归测试。
- run 失败链路已补齐最小失败报告与错误信息保留机制，且报告摘要开始直接使用最后一步的 `progress_summary`，更便于回放失败原因。
- 规则验证护栏已经接在 `validate_agent` 之后，能够对重复等待、重复动作、卡住、偏航和恢复候选做统一收敛；本轮又在 `recovery_candidate` 命中后接入等待观察节点，用于先判断页面是否已经进入可继续状态。
- `wait_observer.py` 新增了独立的等待观察循环，`run_graph` 会在验证后通过 `wait_after_action` 收集等待判断轨迹，并把 `wait_observation_status`、`wait_observation_reason`、`wait_observation_observations` 与 `wait_observation_traces` 写入步骤日志。
- 当前 `RunReport` 已增加 `conclusion` 字段，且结论值固定为 `keep` / `optimize` / `fix`；`report_builder` 会结合真实步骤、摩擦信号、最后一步结论、persona 与 task 生成更详细的 `key_findings`，成功且无明显问题时允许 `next_recommendations` 为空。
- 项目事实文档目录已统一迁移到仓库根目录 `.claude_introduction/`，`.claude` 内技能、agent、规则与 query-project 默认索引目录都需要继续以该路径为准，避免引用旧目录导致检索或说明失效。
- 基础 RAG 上下文层已完成最小闭环，`load_demo_context` 会注入 run 级 `retrieval_context`，`decide` / `validate` prompt、`StepLog` 与 `RunReport` 都已接入同一份检索上下文。
- 动作模型已从旧的 `target/value` 平面字段收口为 `ActionInput.action + payload`，并由 `ACTION_REGISTRY` 统一渲染 prompt 与校验全部 15 种受控动作的分动作 payload（navigate/click/fill/wait/press/scroll/upload/select/hover/check/uncheck/dblclick/drag/ask_for_help/abandon）；执行器、验证器、报告和回归测试均按 payload 消费动作参数。
- 当前项目仍缺少更完整的 UI 经验库、正式持久化与结构化摩擦分析，后续扩展需要继续沿现有 demo run 主链路增量接入。
- 异常补报告路径已与正常收尾路径对齐，异常中断时也会写入 `conclusion="fix"`，避免报告结构不一致。

## 后续目标调整

- 下一阶段优先把等待观察链路升级为真正的受控恢复动作，复用现有验证护栏输出的 `recovery_candidate` 边界。
- 持久化替换应优先兼容现有 `run_store` 接口，减少对 `demo_run_graph` 主流程的破坏。
- RAG、失败恢复和摩擦分析建议沿现有 observe / decide / validate / log 主链路增量接入。
