# 核心数据流

## Demo Run 为等待观察恢复链路补充受控恢复动作与步骤明细透出

### 本次更新原因

- 关联 Commit：无
- 更新原因：本次提交把等待观察中的 `recovery_candidate` 从“继续等待或直接失败”调整为“先记录等待观察结果，再执行一次受控恢复动作”。同时为了让 `/steps` 与 `/report` 能完整还原恢复分支，本次同步补充了 `before_page_state`、`after_page_state` 与 `step_details` 的序列化透出，需要更新 run graph、state/schema、报告构建和回归验证说明。

### 链路目标

在保留 `/runs/demo/*` 现有 API 入口的前提下，让等待观察命中恢复信号时先形成独立 wait step，再由 graph 自动构造一次受控 `navigate` 恢复动作回到 `task.start_url` 并继续主链路；同时把恢复前后页面快照、等待观察逐轮截图和报告级步骤明细统一暴露给 `/steps` 与 `/report`，保证恢复链路可追踪、可验证、可解释。

### 修改前数据流

`validate_progress` 在等待观察结束后，只有 `stuck_page + recovery_candidate` 才会重新进入 `wait_after_action`。`abnormal_stuck` 一旦到达验证节点，通常会直接归一化为失败结论并结束 run，没有独立的恢复动作步骤。`RunState` 也没有记录是否已经尝试过恢复，因此无法区分“首次卡住，应该恢复”和“恢复后仍卡住，应该停止”。另外，`StepLog` 和 `RunReport` 虽然已经保留动作前后页面快照，但 `/report` 不会额外输出结构化 `step_details`，恢复分支的页面证据无法直接从最终报告读取。

### 修改后数据流

`POST /api/v1/runs/demo/start` → `backend.api.routes.demo_runs.start_demo_run()` → `backend.graph.demo_run_graph.create_demo_placeholder_record()` → `asyncio.create_task(run_demo_workflow(...))` → `backend.graph.demo_run_graph.run_demo_workflow()` → `backend.graph.run_graph.run_workflow(..., load_demo_context)` → `load_demo_context` → `init_session` → `observe_page` → `decide_action` → `execute_action` → `wait_after_action` / `observe_after_action` → `validate_progress(首次 abnormal_stuck 产出 recovery_candidate)` → `log_step(先落等待观察 step)` → `prepare_recovery_action(构造 navigate 回起始页)` → `execute_action` → `observe_after_action` → `validate_progress(恢复后仍卡住则失败，否则继续)` → `log_step(落恢复动作 step)` → `observe_page` / `finalize_report(输出 step_details)` → `backend.stores.in_memory_run_store`

### 输入

- `RunRequest`
  - 来源：用户 / API
  - 关键字段：`run_name`、`headless`
  - 约束：由 `POST /runs/demo/start` 接收；`headless` 为空时回退到系统默认浏览器配置
- `ValidationResult`
  - 来源：等待观察后的验证节点 / 验证 agent
  - 关键字段：`status`、`should_stop`、`progress_summary`、`friction_signals`
  - 约束：当 `friction_signals` 包含 `recovery_candidate` 时，graph 会优先继续等待或准备恢复；若已恢复过一次，再次命中 `abnormal_stuck` 时必须转为失败
- `RunState.recovery_attempted`
  - 来源：graph 内部状态
  - 关键字段：布尔值
  - 约束：首次受控恢复前为 `False` 或缺省；`prepare_recovery_action` 执行后写为 `True`，用于阻止无限恢复循环
- `StepLog`
  - 来源：`log_current_step()`
  - 关键字段：`observed_page_state`、`post_action_page_state`、`wait_observation_traces`
  - 约束：通过计算字段额外暴露 `before_page_state` 与 `after_page_state`；API 返回和报告序列化都复用同一份结构化步骤数据

### 处理步骤

1. `backend/api/routes/demo_runs.py::start_demo_run`
   - 职责：保留 demo API 入口，创建占位记录并异步启动后台 run
   - 调用方：HTTP 客户端
   - 被调用方：`get_settings()`、`create_demo_placeholder_record()`、`run_store.create_run()`、`run_demo_workflow()`
   - 调用时机：收到 `POST /runs/demo/start` 时
   - 输入：`Request`、`RunRequest`
   - 输出：包含 `run_id` 与 `queued` 状态的 `ApiResponse`
   - 状态变化：在内存存储中创建 `RunRecord(status=queued)`
   - 异常分支：创建后续流程失败时由后台 run 负责落失败状态；接口自身仅处理同步阶段异常

2. `backend/graph/demo_run_graph.py::load_demo_context`
   - 职责：在真正执行前加载 demo 场景的默认 `Persona`、`Task`、agent 与运行记录
   - 调用方：`backend.graph.run_graph.build_run_graph()` 生成的 LangGraph
   - 被调用方：`build_demo_persona()`、`build_demo_task()`、`create_run_agents()`、`run_store.create_run()`、`run_store.mark_running()`
   - 调用时机：通用 graph 启动后的第一个节点
   - 输入：`RunState`
   - 输出：包含 `persona`、`task`、`record`、`decide_agent`、`validate_agent`、`wait_agent` 的状态增量
   - 状态变化：把运行记录切为 `running`，并把 demo 上下文注入共享状态
   - 异常分支：缺失必要状态时中断 graph，由通用异常兜底补失败报告

3. `backend/graph/run_graph.py::route_after_validate`
   - 职责：根据本轮验证结果判断是直接写 step，还是进入等待观察
   - 调用方：LangGraph 条件边
   - 被调用方：无
   - 调用时机：`validate_current_progress()` 之后
   - 输入：`current_validation_result`、`wait_observation_status`
   - 输出：`wait_after_action` 或 `log_step`
   - 状态变化：当 `friction_signals` 含有 `recovery_candidate` 时，无论触发来源是 `stuck_page`、`repeated_wait`、`repeated_action_target` 还是 `off_track_navigation`，都会继续进入等待观察
   - 异常分支：已存在 `wait_observation_status` 时不再重复进入等待观察，避免同一轮重复等待

4. `backend/graph/run_graph.py::validate_current_progress`
   - 职责：把等待观察结果归一化为继续运行、准备恢复或最终失败
   - 调用方：LangGraph
   - 被调用方：`validate_agent.ainvoke()`、`backend.analysis.validator.validate_progress()`、内部 `_build_wait_*_validation()`
   - 调用时机：非 wait 动作执行后，或 `wait_after_action` 结束后
   - 输入：`task`、`current_execution_result`、`current_page_state`、`step_logs`、`wait_observation_status`、`recovery_attempted`
   - 输出：`ValidationResult`
   - 状态变化：首次命中 `abnormal_stuck` 且尚未恢复过时，返回 `status=running` 且带 `wait_observe_abnormal_stuck + recovery_candidate`；恢复后再次命中 `abnormal_stuck` 则改为 `status=failed`、`should_stop=True`
   - 异常分支：缺失验证前置状态时直接失败；普通等待超时仍按失败处理

5. `backend/graph/run_graph.py::log_current_step`
   - 职责：先把等待观察 step 或恢复动作 step 写入 `StepLog`
   - 调用方：LangGraph
   - 被调用方：`run_store.add_step()`
   - 调用时机：每轮验证完成后
   - 输入：`step_before_page_state`、`post_action_page_state`、`current_action`、`current_execution_result`、`current_validation_result`
   - 输出：更新后的 `step_logs` 与 `current_step_index`
   - 状态变化：先留下一条包含等待观察详情的 step，再为后续受控恢复动作生成新的 step_index
   - 异常分支：任一必需字段缺失时直接失败

6. `backend/graph/run_graph.py::route_after_log`
   - 职责：决定本轮写完 step 后是继续主链路、结束 run，还是转入恢复动作准备节点
   - 调用方：LangGraph 条件边
   - 被调用方：无
   - 调用时机：`log_current_step()` 之后
   - 输入：`should_stop`、`wait_observation_status`、`current_validation_result`
   - 输出：`finalize_report`、`prepare_recovery_action` 或 `observe_page`
   - 状态变化：等待观察命中 `abnormal_stuck + recovery_candidate` 时，不再直接回到 `observe_page`，而是切换到受控恢复分支
   - 异常分支：若 `should_stop=True` 则优先结束，避免失败结论后再误触发恢复

7. `backend/graph/run_graph.py::prepare_recovery_action`
   - 职责：构造最小受控恢复动作，并清空上一轮等待观察残留状态
   - 调用方：`route_after_log()` 命中的恢复分支
   - 被调用方：无
   - 调用时机：等待观察 step 已落库且判定需要恢复时
   - 输入：`task.start_url`、`current_page_state`
   - 输出：新的 `current_action=ActionInput(action="navigate", target=task.start_url, ...)`
   - 状态变化：写入 `recovery_attempted=True`，并清空 `wait_observation_status`、`wait_observation_reason`、`wait_observation_traces` 等等待观察字段，防止恢复动作复用旧状态
   - 异常分支：缺失 `task` 或 `current_page_state` 时直接抛错，中断当前 run

8. `backend/graph/run_state.py::RunState`
   - 职责：为 graph 增加恢复状态位
   - 调用方：`run_graph` 各节点
   - 被调用方：无
   - 调用时机：graph 状态定义阶段
   - 输入：无
   - 输出：增加 `recovery_attempted`
   - 状态变化：为等待观察到恢复分支提供一次性控制条件
   - 异常分支：无

9. `backend/schemas/run_schemas.py::StepLog` / `RunReport`
   - 职责：统一透出恢复链路所需的步骤明细
   - 调用方：`/steps`、`/report` 序列化链路
   - 被调用方：Pydantic 计算字段与模型序列化
   - 调用时机：步骤返回和报告返回时
   - 输入：`StepLog`、`RunReport`
   - 输出：`before_page_state`、`after_page_state`、`step_details`
   - 状态变化：`StepLog` 通过计算字段把 `observed_page_state` / `post_action_page_state` 映射为 API 可读字段；`RunReport` 增加结构化 `step_details`
   - 异常分支：字段缺失时由模型校验失败阻断输出

10. `backend/analysis/report_builder.py::build_run_report`
   - 职责：把完整步骤序列写入最终报告
   - 调用方：`finalize_report()`
   - 被调用方：`_serialize_step()`
   - 调用时机：run 结束收尾时
   - 输入：`RunRecord`、`list[StepLog]`
   - 输出：包含 `step_details` 的 `RunReport`
   - 状态变化：最终报告不再只给摘要，还能直接回放恢复前后页面证据和等待观察 trace
   - 异常分支：步骤序列化失败时中断报告构建

11. `tests/test_validator.py` / `tests/test_demo_run_api.py`
   - 职责：回归验证恢复候选路由、受控恢复动作与 API 字段透出
   - 调用方：pytest
   - 被调用方：`run_graph`、`build_run_report()`、demo API 路由
   - 调用时机：测试运行时
   - 输入：构造的 `RunState`、`StepLog`、`RunRecord`
   - 输出：对 `route_after_validate()`、`validate_current_progress()`、`route_after_log()`、`prepare_recovery_action()` 以及 `/steps` `/report` 返回字段的断言结果
   - 状态变化：无
   - 异常分支：本地环境在测试收集阶段可能被 Anaconda 的 torch `c10.dll` 加载失败阻断

### 状态流转

- 成功路径：`queued` → `running` → `wait_observation_recorded` → `recovery_navigate_started` → `running` / `succeeded`
- 失败路径：`queued` → `running` → `wait_observation_recorded` → `recovery_navigate_started` → `failed`

### 输出

- `list[StepLog]`
  - 去向：`GET /runs/demo/{run_id}/steps`
  - 用途：查看等待观察 step 与恢复动作 step 各自的 `before_page_state`、`after_page_state`、`wait_observation_traces[].screenshot_path` 和恢复动作类型
- `RunReport`
  - 去向：`GET /runs/demo/{run_id}/report`
  - 用途：查看最终状态、摩擦信号、结构化 `step_details`，并直接核对恢复前后页面证据与等待观察截图

### 错误分支

- 首次异常卡住
  - 触发条件：`wait_after_action` 返回 `wait_observation_status="abnormal_stuck"`，且 `recovery_attempted` 为空或为 `False`
  - 处理方式：`validate_current_progress()` 返回带 `recovery_candidate` 的运行中结论；`log_current_step()` 先落一条等待观察 step，随后 `prepare_recovery_action()` 构造受控 `navigate`
  - 影响范围：run 不会立即失败，而是新增一条恢复动作 step 继续执行
- 恢复后再次异常卡住
  - 触发条件：`wait_observation_status="abnormal_stuck"` 且 `recovery_attempted=True`
  - 处理方式：验证节点直接返回失败结论，`route_after_log()` 走 `finalize_report`
  - 影响范围：run 终止，但 `/steps` 与 `/report` 会保留完整恢复证据
- 恢复准备前置状态缺失
  - 触发条件：`prepare_recovery_action()` 缺少 `task` 或 `current_page_state`
  - 处理方式：抛出 `ValueError("Task or page state is missing before recovery action.")`
  - 影响范围：当前 run 进入异常收尾，无法执行恢复 navigate
- 报告步骤明细缺失
  - 触发条件：`build_run_report()` 未收到可序列化 `StepLog` 列表
  - 处理方式：报告构建中断，需由调用方回退到通用失败收尾
  - 影响范围：`/report` 无法返回本次 run 的结构化恢复链路
