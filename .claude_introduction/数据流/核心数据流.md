# 核心数据流

## 项目总览

### 本次更新原因(每次只用写最新的一次)

- 关联 Commit：无
- 更新原因：T-022 已把受控动作模型从 4 种扩展到 15 种（新增 press/scroll/upload/select/hover/check/uncheck/dblclick/drag/ask_for_help/abandon），引入 `TERMINAL_ACTIONS` 概念，并同步执行器、验证器、graph 路由、报告和 prompt 渲染链路，需要把当前项目总览改成最新事实。

### 链路目标

说明当前 Synthetic User Lab 在 demo 形态下，如何从 FastAPI 入口启动一次 run，并把页面观察、动作决策、真实执行、等待观察、验证护栏、步骤日志和最终报告串成可回放闭环。

### 修改前数据流

原文档主要记录某次提交改了什么，缺少系统总览、统一入口、共享状态对象和 API 到 graph/store/report 的完整调用关系。

### 修改后数据流

`FastAPI demo run API` → `backend/api/routes/demo_runs.py(_track_background_task)` → `backend/graph/demo_run_graph.py` → `backend/graph/run_graph.py(create_run_agents / finalize_report / run_workflow)` → `Playwright + graph agents + validator + wait_observer + observer` → `backend/stores/in_memory_run_store.py` → `backend/analysis/report_builder.py(build_run_report_async / build_run_report_without_llm)` → `状态 / 步骤 / 报告 API`

### 输入

- `RunRequest`
  - 来源：用户 / API
  - 关键字段：`run_name`、`headless`
  - 约束：当前只支持 demo run；`headless` 为空时回退到服务默认浏览器配置
- `Request.base_url`
  - 来源：FastAPI 请求上下文
  - 关键字段：应用基准 URL
  - 约束：用于生成 demo task 的 `start_url`
- `screenshot_dir`
  - 来源：服务配置
  - 关键字段：截图目录
  - 约束：graph 内所有动作前后截图、等待观察截图都写到该目录下的 `run_id` 子目录

### 处理步骤

1. `backend/api/routes/demo_runs.py::start_demo_run`
   - 职责：接收启动请求、生成 `run_id`、创建占位记录并异步发起后台 run
   - 调用方：HTTP 客户端
   - 被调用方：`create_demo_placeholder_record()`、`run_store.create_run()`、`run_demo_workflow()`、`_track_background_task()`
   - 调用时机：收到 `POST /api/v1/runs/demo/start` 时
   - 输入：`Request`、`RunRequest`
   - 输出：`{"run_id": ..., "status": "queued"}`
   - 状态变化：内存中创建 `RunRecord(status="queued")`，并把后台 task 加入强引用集合等待回调收尾
   - 异常分支：同步阶段异常直接返回 HTTP 错误
2. `backend/graph/demo_run_graph.py::run_demo_workflow`
   - 职责：把 demo persona、task、agent 和配置接到通用 run graph 上执行
   - 调用方：`start_demo_run()` 创建的后台任务
   - 被调用方：`run_workflow()`
   - 调用时机：占位记录创建后
   - 输入：`run_id`、`request`、`app_base_url`、`screenshot_dir`
   - 输出：graph 执行结果
   - 状态变化：由 graph 内部节点推进 run 状态
   - 异常分支：交由 `run_workflow()` 统一失败收尾
3. `backend/stores/in_memory_run_store.py::InMemoryRunStore`
   - 职责：保存当前 demo 运行态的 `RunRecord`、`StepLog[]` 和 `RunReport`
   - 调用方：API 层、graph 节点、报告节点
   - 被调用方：无
   - 调用时机：创建 run、标记运行中、追加步骤、写入报告、查询结果时
   - 输入：`RunRecord`、`StepLog`、`RunReport`
   - 输出：状态、步骤、报告查询结果
   - 状态变化：更新 `_records`、`_steps`、`_reports`
   - 异常分支：当前为进程内存存储，进程重启后数据不保留
4. `backend/analysis/report_builder.py::build_run_report_async`
   - 职责：把步骤证据收敛为 run 级结构化报告，并在主链路中使用异步模型分析避免阻塞事件循环
   - 调用方：`finalize_report()`、异常收尾逻辑
   - 被调用方：`_build_report_base()`、`_generate_report_analysis_async()`、`build_run_report_without_llm()`
   - 调用时机：run 结束时或异常兜底时
   - 输入：`RunRecord`、`StepLog[]`
   - 输出：`RunReport`
   - 状态变化：无
   - 异常分支：建议生成失败时降级为空分析；graph 异常收尾时改走不依赖 LLM 的兜底报告

### 状态流转

- 成功路径：`queued` → `running` → `succeeded`
- 失败路径：`queued` → `running` → `failed`

### 输出

- `RunStatusResponse`
  - 去向：`GET /api/v1/runs/demo/{run_id}`
  - 用途：返回当前运行状态与错误摘要
- `list[StepLog]`
  - 去向：`GET /api/v1/runs/demo/{run_id}/steps`
  - 用途：回放完整步骤证据
- `RunReport`
  - 去向：`GET /api/v1/runs/demo/{run_id}/report`
  - 用途：查看最终结论、关键发现、建议和步骤明细

### 错误分支

- 后台任务提前结束但未写回最终状态
  - 触发条件：`run_demo_workflow()` 的后台 task 被取消或抛出未捕获异常
  - 处理方式：`_track_background_task()` 保持强引用，并在 done callback 中调用 `run_store.fail_run()`
  - 影响范围：避免 run 长时间停留在 `queued/running` 且报告不可读
- `run_id` 不存在
  - 触发条件：查询不存在的运行记录
  - 处理方式：API 返回 `404`
  - 影响范围：状态、步骤、报告接口均不可用
- 报告尚未生成
  - 触发条件：run 未结束就访问 `/report`
  - 处理方式：API 返回 `409`
  - 影响范围：调用方只能稍后重试
- 服务进程重启
  - 触发条件：当前内存 store 被清空
  - 处理方式：当前没有持久化恢复能力
  - 影响范围：历史状态、步骤和报告全部失效

### 相关文件

- `backend/core/config.py`：延迟加载环境变量与运行配置
- `backend/ai_api/provider_router.py`：按需分配模型路由与供应商配置
- `backend/api/routes/demo_runs.py`：demo run API 入口与后台任务追踪
- `backend/graph/demo_run_graph.py`：demo persona/task/agent 装配
- `backend/graph/run_graph.py`：通用 run graph 主流程
- `backend/stores/in_memory_run_store.py`：内存状态存储
- `backend/analysis/report_builder.py`：最终报告构建与 LLM 降级路径
- `backend/schemas/run_schemas.py`：共享数据结构

### 验证方式

- `python -m py_compile backend/graph/run_graph.py backend/prompt/graph.py tests/test_validator.py`：通过
- `python -m pytest tests/test_validator.py -q`：通过，40 passed

## 主要入口与 API → graph → store/report 调用链

### 本次更新原因(每次只用写最新的一次)

- 关联 Commit：无
- 更新原因：T-020 让启动入口需要追踪后台 task，并把异常报告兜底与查询行为改成可读取失败状态，需同步 API 到 graph/store/report 的真实调用链。

### 链路目标

明确 API 层如何启动 run、读取状态、读取步骤和读取报告，以及这些入口与 graph、store、report 的真实调用关系。

### 修改前数据流

原文档没有把启动入口和三个查询入口放在统一调用链中描述，读写边界不清晰。

### 修改后数据流

`POST /runs/demo/start` → `create_demo_placeholder_record` → `run_store.create_run` → `asyncio.create_task` → `_track_background_task` → `run_demo_workflow` → `run_workflow`

`GET /runs/demo/{id}` → `run_store.get_status`

`GET /runs/demo/{id}/steps` → `run_store.get_steps`

`GET /runs/demo/{id}/report` → `run_store.get_report` / `run_store.get_record`

### 输入

- `run_id`
  - 来源：路径参数
  - 关键字段：运行记录唯一标识
  - 约束：必须已存在于 `run_store`
- `RunRecord`
  - 来源：`create_demo_placeholder_record()`、`load_demo_context()`
  - 关键字段：`status`、`request`、`persona`、`task`、`error_type`、`error_message`
  - 约束：API 查询状态和最终报告都依赖它的最终值
- `RunReport`
  - 来源：`build_run_report()` 或异常收尾逻辑
  - 关键字段：`summary`、`conclusion`、`step_details`、`error_type`、`error_message`
  - 约束：只有 run 结束后才能稳定读取

### 处理步骤

1. `backend/api/routes/demo_runs.py::start_demo_run`
   - 职责：创建 `queued` 记录并触发后台 run
   - 调用方：HTTP 客户端
   - 被调用方：`create_demo_placeholder_record()`、`run_store.create_run()`、`asyncio.create_task()`、`_track_background_task()`、`run_demo_workflow()`
   - 调用时机：启动 run 时
   - 输入：`RunRequest`、`Request.base_url`
   - 输出：返回 run 已排队的响应
   - 状态变化：新增 `RunRecord`、空步骤列表与后台 task 强引用
   - 异常分支：同步创建失败直接返回 HTTP 错误
2. `backend/api/routes/demo_runs.py::get_demo_run_status`
   - 职责：返回状态摘要
   - 调用方：HTTP 客户端
   - 被调用方：`run_store.get_status()`
   - 调用时机：查询状态时
   - 输入：`run_id`
   - 输出：`RunStatusResponse`
   - 状态变化：无
   - 异常分支：不存在时返回 `404`
3. `backend/api/routes/demo_runs.py::get_demo_run_steps`
   - 职责：返回完整步骤序列
   - 调用方：HTTP 客户端
   - 被调用方：`run_store.get_steps()`
   - 调用时机：查询步骤时
   - 输入：`run_id`
   - 输出：`list[StepLog]`
   - 状态变化：无
   - 异常分支：不存在时返回 `404`
4. `backend/api/routes/demo_runs.py::get_demo_run_report`
   - 职责：返回最终报告
   - 调用方：HTTP 客户端
   - 被调用方：`run_store.get_report()`、`run_store.get_record()`
   - 调用时机：查询报告时
   - 输入：`run_id`
   - 输出：`RunReport`
   - 状态变化：无
   - 异常分支：不存在时返回 `404`，未完成时返回 `409`
5. `backend/graph/run_graph.py::run_workflow`
   - 职责：统一执行 graph，并在异常时补齐失败状态和失败报告
   - 调用方：`run_demo_workflow()`
   - 被调用方：`build_run_graph()`、`graph.ainvoke()`、`run_store.fail_run()`、`build_run_report_without_llm()`、`run_store.complete_run()`
   - 调用时机：后台 run 开始时
   - 输入：初始 `RunState`
   - 输出：graph 最终状态或抛出异常
   - 状态变化：异常时也会确保 `RunRecord.status` 与 `RunReport` 收敛为失败
   - 异常分支：区分 `model_error` 与 `system_error`，并在报告模型不可用时退回无 LLM 报告构建

### 状态流转

- 成功路径：`create_run` → `mark_running` → `add_step*` → `complete_run` → `GET /report`
- 失败路径：`create_run` → `mark_running` → `fail_run` → `complete_run` → `GET /report`

### 输出

- `ApiResponse`
  - 去向：所有 demo run API
  - 用途：统一 HTTP 返回结构
- `RunReport.step_details`
  - 去向：报告接口
  - 用途：给调用方回放完整 run 证据链

### 错误分支

- 后台 graph 执行异常或任务被取消
  - 触发条件：模型调用错误、浏览器异常、节点前置状态缺失，或 `asyncio` task 被取消
  - 处理方式：`run_workflow()` 统一写入失败状态与失败报告，`_track_background_task()` 在任务未正常收尾时兜底 `fail_run`
  - 影响范围：run 结束为失败，但 `/report` 仍可读取失败摘要
- 报告提前读取
  - 触发条件：run 仍在执行中
  - 处理方式：返回 `409`
  - 影响范围：调用方需轮询状态或稍后重试

### 相关文件

- `backend/api/routes/demo_runs.py`：API 启动与查询入口
- `backend/graph/demo_run_graph.py`：demo graph 启动入口
- `backend/graph/run_graph.py`：统一执行与异常收尾
- `backend/stores/in_memory_run_store.py`：读写状态事实源
- `backend/analysis/report_builder.py`：报告构建

### 验证方式

- `python -m py_compile backend/graph/run_graph.py backend/prompt/graph.py tests/test_validator.py`：通过
- `python -m pytest tests/test_validator.py -q`：通过，40 passed

## Demo Run 主链路

### 本次更新原因(每次只用写最新的一次)

- 关联 Commit：无
- 更新原因：T-020 调整了 graph 模型初始化、报告异步收尾、失败兜底与 selector 输出约束，需要把主链路节点职责改成当前实现事实。

### 链路目标

让固定 demo persona、固定 demo task 和内置 demo 页面形成一次可验证、可解释、可回放的最小纵向闭环。

### 修改前数据流

原文档没有集中描述主链路节点顺序、节点职责和节点间共享状态依赖。

### 修改后数据流

`load_demo_context` → `init_session` → `observe_page` → `decide_action` → `execute_action` → `observe_after_action / wait_after_action` → `validate_progress` → `log_step` → `observe_page / prepare_recovery_action / finalize_report`

### 输入

- `Persona`
  - 来源：`build_demo_persona()`
  - 关键字段：`name`、`skill_level`、`patience_level`、`risk_preference`
  - 约束：当前为固定 demo persona
- `Task`
  - 来源：`build_demo_task()`
  - 关键字段：`start_url`、`success_criteria`、`max_steps`、`allowed_actions`
  - 约束：当前 demo task 允许全部 15 种受控动作（navigate/click/fill/wait/press/scroll/upload/select/hover/check/uncheck/dblclick/drag/ask_for_help/abandon），其中 ask_for_help 和 abandon 为终止动作
- `RunState`
  - 来源：`run_workflow()` 初始化
  - 关键字段：`run_id`、`request`、`app_base_url`、`screenshot_dir`
  - 约束：后续节点都以增量方式更新同一份图状态

### 处理步骤

1. `backend/graph/demo_run_graph.py::load_demo_context`
   - 职责：构造 demo persona、task 和三类 agent，并把 run 标记为运行中
   - 调用方：LangGraph
   - 被调用方：`build_demo_persona()`、`build_demo_task()`、`create_run_agents()`、`run_store.mark_running()`
   - 调用时机：graph 启动后的第一个节点
   - 输入：`run_id`、`request`、`app_base_url`
   - 输出：`persona`、`task`、`record`、`decide_agent`、`validate_agent`、`wait_agent`、`retrieval_context`
   - 状态变化：`RunRecord.status` 从 `queued` 变为 `running`，并在节点内完成 settings / model / checkpointer 的延迟创建
   - 异常分支：必要状态缺失时中断 graph
2. `backend/graph/run_graph.py::init_session`
   - 职责：创建 Playwright 会话并进入任务起始页
   - 调用方：LangGraph
   - 被调用方：`create_browser_session()`
   - 调用时机：上下文加载后
   - 输入：`task.start_url`、`request.headless`
   - 输出：`session`
   - 状态变化：共享状态写入浏览器会话
   - 异常分支：浏览器初始化失败时进入异常收尾
3. `backend/graph/run_graph.py::observe_current_page`
   - 职责：记录当前页面事实
   - 调用方：LangGraph
   - 被调用方：`backend.execution.observer.observe_page()`
   - 调用时机：进入主链路首轮或每轮继续前
   - 输入：`session.page`
   - 输出：`current_page_state`
   - 状态变化：刷新当前页面快照，输出可执行 selector 并对 `id/name` 特殊字符做转义
   - 异常分支：页面观察失败时终止当前 run
4. `backend/graph/run_graph.py::decide_next_action`
   - 职责：基于页面事实、最近步骤和 persona/task 选择下一步受控动作
   - 调用方：LangGraph
   - 被调用方：`decide_input_prompt.format_messages()`、`decide_agent.ainvoke()`
   - 调用时机：每轮页面观察后
   - 输入：`current_page_state`、`step_logs`、`retrieval_context`
   - 输出：`current_action`
   - 状态变化：共享状态写入本轮动作
   - 异常分支：模型 API 错误或格式错误超限后进入异常收尾
5. `backend/graph/run_graph.py::execute_current_action`
   - 职责：记录动作前截图并执行受控动作
   - 调用方：LangGraph
   - 被调用方：`execute_action()`、`page.screenshot()`
   - 调用时机：动作决策后
   - 输入：`current_action`、`current_page_state`
   - 输出：`current_execution_result`、`step_before_page_state`
   - 状态变化：清空上一轮等待观察残留字段
   - 异常分支：动作执行失败会在验证节点被收敛为失败
6. `backend/graph/run_graph.py::route_after_execute`
   - 职责：把 `wait` 动作直接路由到等待观察节点，终止动作（ask_for_help/abandon）直接路由到验证节点，其他动作先观察动作后页面
   - 调用方：LangGraph 条件边
   - 被调用方：无
   - 调用时机：动作执行后
   - 输入：`current_action`
   - 输出：`wait_after_action` 或 `observe_after_action`
   - 状态变化：无
   - 异常分支：无
7. `backend/graph/run_graph.py::observe_after_action`
   - 职责：记录动作后页面快照与 after 截图
   - 调用方：LangGraph
   - 被调用方：`observe_page()`、`page.screenshot()`
   - 调用时机：非 wait 动作执行后
   - 输入：`session.page`
   - 输出：`current_page_state`、`post_action_page_state`
   - 状态变化：共享状态刷新动作后页面
   - 异常分支：观察失败时当前 run 结束
8. `backend/graph/run_graph.py::validate_current_progress`
   - 职责：把动作结果、动作后页面、历史步骤和任务成功标准收敛为最终验证结论
   - 调用方：LangGraph
   - 被调用方：`validate_input_prompt.format_messages()`、`validate_agent.ainvoke()`、`backend.analysis.validator.validate_progress()`
   - 调用时机：动作后观察完成后，或等待观察终止后
   - 输入：`current_execution_result`、`current_page_state`、`task.success_criteria`、`step_logs`、`retrieval_context`
   - 输出：`current_validation_result`、`should_stop`
   - 状态变化：共享状态记录本轮是否需要停止，并保证模型已判定成功时不会再被步数上限覆盖
   - 异常分支：缺失验证前置状态时直接失败
9. `backend/graph/run_graph.py::log_current_step`
   - 职责：沉淀单步执行证据
   - 调用方：LangGraph
   - 被调用方：`run_store.add_step()`
   - 调用时机：本轮验证完成后
   - 输入：动作前后页面、动作、执行结果、验证结论、等待观察细节、检索上下文

### 链路目标

### 修改前数据流

### 错误分支

- 共享状态残留污染下一步

  - 处理方式：`execute_current_action()` 和 `prepare_recovery_action()` 会清空相关残留字段
  - 影响范围：避免日志和路由混入上一轮等待结果

## 基础 RAG 上下文层

### 本次更新原因(每次只用写最新的一次)

- 关联 Commit：无
- 更新原因：T-021 新增了最小 RAG 上下文层，需要把产品知识与失败案例检索结果从 run 初始化一路透传到决策、验证、步骤日志和最终报告，避免文档仍停留在“仅有主链路”的旧状态。

### 链路目标

在 demo run 中为决策与验证节点提供可复用的产品知识与失败案例上下文，并将同一份检索结果沉淀到 `StepLog` 与 `RunReport` 中，保证报告证据和决策依据一致。

### 修改前数据流

新增链路，修改前无。

### 修改后数据流

`build_retrieval_context()` → `load_demo_context()` → `RunState.retrieval_context` → `decide_next_action()` / `validate_current_progress()` → `log_current_step()` → `RunReport`

### 输入

- `Persona` / `Task`
  - 来源：`build_demo_persona()`、`build_demo_task()`
  - 关键字段：persona 行为偏好、任务目标与动作边界
  - 约束：只用于构建 run 级检索上下文，不替代实时页面事实
- `retrieval_context`
  - 来源：`backend/retrieval/minimal_context.py::build_retrieval_context()`
  - 关键字段：`source_type`、`title`、`content`、`source_ref`
  - 约束：当前只覆盖产品知识与失败案例两类片段

### 处理步骤

1. `backend/retrieval/minimal_context.py::build_retrieval_context`
   - 职责：基于 persona 与 task 生成最小检索上下文
   - 调用方：`load_demo_context()`
   - 被调用方：无
   - 调用时机：run 初始化阶段
   - 输入：`Persona`、`Task`
   - 输出：`list[RetrievedContextItem]`
   - 状态变化：无
   - 异常分支：无外部依赖，返回空列表或最小默认片段
2. `backend/graph/demo_run_graph.py::load_demo_context`
   - 职责：把检索上下文装配进 run 级共享状态
   - 调用方：LangGraph
   - 被调用方：`build_retrieval_context()`
   - 调用时机：demo graph 启动时
   - 输入：`persona`、`task`
   - 输出：`retrieval_context`
   - 状态变化：`RunState` 持有本次 run 的检索上下文
   - 异常分支：无
3. `backend/graph/run_graph.py::decide_next_action` / `validate_current_progress`
   - 职责：在生成 prompt 时渲染检索上下文，供决策与验证模型参考
   - 调用方：LangGraph
   - 被调用方：`render_retrieval_context()`、`decide_input_prompt.format_messages()`、`validate_input_prompt.format_messages()`
   - 调用时机：每轮动作决策与验证前
   - 输入：`retrieval_context`
   - 输出：带检索上下文的提示词消息
   - 状态变化：无
   - 异常分支：无
4. `backend/graph/run_graph.py::log_current_step`
   - 职责：把检索上下文写入步骤日志
   - 调用方：LangGraph
   - 被调用方：`run_store.add_step()`
   - 调用时机：每轮验证后
   - 输入：`retrieval_context`
   - 输出：`StepLog.retrieval_context`
   - 状态变化：步骤证据增加可回放的检索依据
   - 异常分支：无
5. `backend/analysis/report_builder.py::_serialize_step`
   - 职责：把步骤日志中的检索上下文序列化进报告
   - 调用方：`build_run_report_async()`
   - 被调用方：无
   - 调用时机：生成 `RunReport` 时
   - 输入：`StepLog.retrieval_context`
   - 输出：`RunReport.step_details[].retrieval_context`
   - 状态变化：无
   - 异常分支：无

### 状态流转

- 成功路径：`build_retrieval_context` → `RunState.retrieval_context` → `decide/validate` → `StepLog.retrieval_context` → `RunReport.step_details.retrieval_context`
- 失败路径：`build_retrieval_context` 为空或仅返回默认片段 → `decide/validate` 仍可只依赖实时页面事实 → `StepLog` 与 `RunReport` 保留实际检索结果

### 输出

- `RunState.retrieval_context`
  - 去向：决策节点、验证节点、日志节点、报告节点
  - 用途：在同一 run 内复用检索上下文
- `StepLog.retrieval_context`
  - 去向：步骤接口、报告接口
  - 用途：保留决策依据与报告证据

### 错误分支

- 检索上下文为空
  - 触发条件：任务或画像不足以构造有效片段
  - 处理方式：仍允许 run 继续，决策与验证退回实时页面事实
  - 影响范围：本次 run 的上下文提示更短，但不阻塞主链路
